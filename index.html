Problem问题

Expert-dependent, manual workflows in a regulated multi-region environment. Every output needed citations, audit trails, and jurisdictional compliance. No existing platform met all three requirements simultaneously. 受监管多区域环境中依赖专家的手动工作流。每个输出都需要引用、审计追踪和司法管辖合规。没有现有平台能同时满足这三项要求。

Key Decisions关键决策

  • Citation as a platform contract — "No evidence, no conclusion." Every output must carry verifiable references with version, jurisdiction, and entitlement metadata. 引用即平台契约——"无证据，无结论"。每个输出必须携带可验证的引用，包含版本、管辖区和权限元数据。
  • Deterministic safety gating — Four-outcome routing (answer / refuse / citations-only / escalate) using rule-based checks, not another LLM call. Keeps control fast and auditable. 确定性安全门控——四路结果路由（回答 / 拒绝 / 仅引用 / 升级），使用基于规则的检查而非 LLM 调用。保持控制快速且可审计。
  • Hybrid retrieval with metadata-aware reranking — Dense vector search (Milvus, 768-dim embeddings from domain fine-tuned model) handles semantic matching; sparse BM25 handles exact terminology — regulatory codes, entity names, jurisdiction identifiers. A cross-encoder reranker scores the merged candidate set, boosted by metadata signals: jurisdiction match, document currency, and entitlement alignment. Chunk strategy is domain-specific: clause-level for regulations (preserving section hierarchy), paragraph-level for internal documents. 混合检索与元数据感知重排序——Dense 向量搜索（Milvus，领域微调模型生成的 768 维 embedding）处理语义匹配；稀疏 BM25 处理精确术语——法规编号、实体名称、管辖区标识。Cross-encoder reranker 对合并候选集评分，叠加元数据信号：管辖区匹配、文档时效性和权限对齐。分块策略按领域定制：法规按条款级切分（保留章节层级），内部文档按段落级切分。
  • Replay-based evaluation — Captured production traces feed offline regression suites. Model updates go through promotion gates before canary rollout. 基于重放的评估——捕获生产 trace 用于离线回归测试。模型更新需通过准入门控后再进行金丝雀发布。

Workflow Automation Agent — Plan-and-Execute Across Internal Systems 工作流自动化 Agent — 跨内部系统的计划执行引擎

Tool CallingLeast-privilegeCircuit BreakersEnd-to-end Traceability

Problem问题

Multi-step workflows spanning 10–20 internal systems. Manual execution was slow, error-prone, and leaked data across system boundaries. No audit trail for cross-system operations. 跨 10–20 个内部系统的多步骤工作流。手动执行缓慢、易出错，且跨系统边界泄露数据。跨系统操作无审计追踪。

Key Decisions关键决策

  • Reusable plan-and-execute engine — Multi-step tasks become governed, replayable pipelines under least-privilege constraints with step budgets and circuit breakers. 可复用的计划执行引擎——多步骤任务变为受治理的、可重放的管道，在最小权限约束下运行，配备步骤预算和熔断器。
  • Shift-left validation gates — Time window, owner, approval chain, authoritative source, and conflict detection checks run before downstream delivery. 质量左移的验证门控——时间窗口、所有者、审批链、权威来源和冲突检测在下游交付前完成检查。
  • Minimum-disclosure output schema — Standardized packages expose only IDs, required fields, timestamps, and hashes. Reduced manual redaction load by ~50%. 最小披露输出模式——标准化包仅暴露 ID、必要字段、时间戳和哈希。手动脱敏负载降低约 50%。

Decision Support Agent — Maker-Checker & Human-in-the-Loop 决策支持 Agent — Maker-Checker 与人在环路

Planner/Executor/CriticMaker-CheckerDecision ArbitrationAuditable Replay

Problem问题

High-risk decision points relied on experience-driven, scattered approvals. No standardized review criteria, no audit trail, and sign-off coverage was only 70–80% for critical decisions. 高风险决策点依赖经验驱动的分散审批。无标准化审查标准、无审计追踪，关键决策的签核覆盖率仅 70–80%。

Key Decisions关键决策

  • Maker-checker workflow — "No sign-off, no issuance." Role-based collaboration with Planner, Executor, and Critic roles, each with codified responsibilities and arbitration rules. Maker-Checker 工作流——"无签核、无签发"。基于角色的协作（Planner / Executor / Critic），每个角色有明确的职责和仲裁规则。
  • Executable review criteria — Codified rules for approve / rework / escalate based on evidence completeness, entitlement blocks, and jurisdiction constraints. Median re-review cycles: 2 → 1. 可执行的审查标准——基于证据完整性、权限阻断和管辖区约束的编码规则（批准 / 返工 / 升级）。中位重审轮次：2 → 1。
