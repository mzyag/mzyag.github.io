One Harness for Every Agent: Unifying Governance Across Tools and Interfaces 一套 Harness 治理所有 Agent：跨工具统一治理规范

If you're running more than one AI agent in production — say an interactive CLI tool alongside an async chatbot or orchestrator — you'll eventually need them to follow the same governance rules. Same risk classification, same confirmation protocol, same reporting format. We hit this wall when our second agent tried to follow a spec that had only ever been tested by one. It broke in three places within the first week. The risk model couldn't express what it needed to. The confirmation gates assumed a user who was always present. The execution record formats were silently incompatible. This is the story of fixing those breaks — and, more honestly, the five-round template iteration where I kept getting the delivery report wrong before checking what Google, React, and ITIL actually do. 如果你同时跑着不止一个 AI Agent——比如一个交互式 CLI 工具加上一个异步 chatbot 或编排平台——迟早要让它们遵循同一套治理规范。同样的风险分类、同样的确认协议、同样的报告格式。当我们的第二个 Agent 尝试遵循一套只被一个 Agent 验证过的规范时，第一周就在三个地方崩了。风险模型表达力不够。确认门控假设用户一直在线。两边的执行记录格式静默不兼容。这篇讲的是修复这些断裂的过程——更坦诚地说，也是我在交付报告模板上迭代了五轮、直到不得不停下来去查 Google、React 和 ITIL 到底怎么做的过程。

Two Agents, One Spec, Zero Alignment两个 Agent，一套规范，零对齐

Some context on what we were running. Agent one: an interactive CLI agent (Claude CLI) that runs locally, user sitting right there, confirmation happens inline in the terminal. I described the original harness design and the memory architecture in earlier posts — five-layer governance, human checkpoints, episode capture for self-improvement. The whole thing assumed real-time interaction. 先说背景。第一个 Agent：交互式 CLI Agent（Claude CLI），在本地运行，用户就坐在旁边，确认在终端里内联完成。我在之前的文章里写了最初的 harness 设计和记忆架构——五层治理、人工检查点、用于自改进的 episode 捕获。整套设计的前提是实时交互。

Agent two: an async multi-agent orchestrator that dispatches tasks to LLM agents via API calls. Someone submits a task at 10 AM, gets pulled into a meeting, comes back at 2 PM and expects to find the result. No "same turn." No local filesystem. Every confirmation needs to be a checkpoint that survives process restarts — because the orchestrator runs server-side, and we restart it for deploys. 第二个 Agent：异步多 Agent 编排平台，通过 API 调用将任务派发给 LLM Agent。有人上午 10 点提交一个任务，开个会，下午 2 点回来期望看到结果。没有"同一轮"。没有本地文件系统。每个确认必须是能挺过进程重启的检查点——因为编排平台跑在服务端，部署时会重启。

Both needed to follow the same governance spec. If "high risk" means different things depending on which agent processed the request, the whole classification system is noise — nobody trusts it, nobody acts on it. But the async agent had evolved its own parallel system over three months: its own checkpoint sequence, a one-dimensional low/medium/high risk model, direct JSONL writes for execution records. Two governance systems, maintained independently, drifting with every commit. If you've ever had two microservices implement the same business rule differently because nobody owned the shared contract — same dynamic, different domain. 两者都需要遵循同一套治理规范。如果"high risk"在不同 Agent 里含义不同，整个分类体系就是噪音——没人信它，没人按它行动。但异步 Agent 在三个月里已经进化出了自己的并行体系：自己的检查点序列、一维的 low/medium/high 风险模型、直接往 JSONL 写执行记录。两套治理系统，各自维护，每次提交都在加大分裂。如果你经历过两个微服务因为没人维护共享契约而把同一条业务规则实现成了两个版本——同样的动态，不同的领域。

I'd been ignoring the divergence because both agents "worked." What forced the issue was the execution record pipeline — the system that records what each agent did, whether it succeeded, and what went wrong, so we can spot patterns and improve over time. We tried to aggregate these records across both agents and discovered that nearly half of the async agent's entries were silently dropped. Different field names — episode_id vs id, task_scope vs scope. Same concept, schema mismatch, zero errors logged. We were missing almost half of one agent's execution history and didn't know it. 我一直忽视这个分歧，因为两个 Agent 都"能用"。真正逼我面对的是执行记录管道——也就是记录每个 Agent 做了什么、成功还是失败、哪里出了问题的系统，用来发现规律和持续改进。我们尝试把两个 Agent 的记录合到一起分析，发现异步 Agent 有将近一半的记录在合并时被悄悄丢掉了。字段名不同——episode_id 对 id，task_scope 对 scope。记的是同一个东西，但格式对不上，程序直接跳过，没有任何报错。将近一半的执行记录丢了，我们却一直不知道。

Risk Classification: From One Dimension to Three风险分类：从一维到三维

The async agent's risk model was a single axis: low | medium | high. The LLM dispatcher picked one based on a prompt that said "consider the complexity and risk." Worked fine for most tasks. The edge cases exposed the problem. 异步 Agent 的风险模型是单一维度：low | medium | high。LLM dispatcher 根据一句"考虑复杂度和风险"的 prompt 来选择。大多数任务没问题。边界情况暴露了问题。

A concrete case: modifying one config file to update a database connection string. Single file — sounds like low risk. But that config pointed to the production database, and a wrong value would break the live service. A senior engineer on the team flagged it after the async agent auto-dispatched it without confirmation. One-dimensional risk can't express "small change, high stakes." It forces the dispatcher to pick one signal and drop the other. 一个具体案例：修改一个配置文件更新数据库连接字符串。单个文件——听起来像低风险。但那个配置指向生产数据库，一个错误值会搞垮线上服务。团队里一个高级工程师在异步 Agent 未经确认就自动派发后把这个问题提了出来。一维风险无法表达"改动小、风险大"。它迫使 dispatcher 选一个信号、丢掉另一个。

We replaced it with a three-axis matrix adapted from OWASP LLM06's excessive-agency taxonomy — the same framework we use for production agent governance: 我们用一个三轴矩阵替换了它，改编自 OWASP LLM06 的过度代理分类——和我们用于生产 Agent 治理的框架一致：

  • Blast radius (L0–L2): 1-2 files → multiple modules → cross-project or external APIs影响面（L0–L2）：1-2 文件 → 多模块 → 跨项目或外部 API
  • Reversibility (L0–L2): git revert → DB migration → external API call or production data可逆性（L0–L2）：git revert → 数据库迁移 → 外部 API 调用或生产数据
  • Sensitivity (L0–L2): UI/docs → business logic → auth/payment/PII敏感度（L0–L2）：UI/文档 → 业务逻辑 → 认证/支付/PII

Derivation: take the max across all three. All L0 = Routine. Any L1 = Standard. Any L2 = Critical. That config file case? Blast radius L0 (one file), reversibility L1 (revertable but service-impacting), sensitivity L2 (production infrastructure). Max is L2, so Critical — agent must get explicit confirmation. No more guessing. 推导：三个轴取最大值。全 L0 = Routine。任一 L1 = Standard。任一 L2 = Critical。那个配置文件的案例？影响面 L0（一个文件），可逆性 L1（可回滚但影响服务），敏感度 L2（生产基础设施）。最大值 L2，所以是 Critical——Agent 必须获得显式确认。不用再猜了。

In the async agent's dispatcher prompt, the output now includes a structured risk_axes object: 在异步 Agent 的 dispatcher prompt 中，输出现在包含结构化的 risk_axes 对象： 

{
  "risk_level": "Critical",
  "risk_axes": {
    "blast_radius": "L0",
    "reversibility": "L1",
    "sensitivity": "L2"
  }
}

Both agents now produce the same classification for the same task. This matters because governance loses credibility the moment two tools disagree on risk for the same change — if "Standard" means different things depending on which agent runs the task, the team stops trusting the classification and starts ignoring it. 两个 Agent 现在对同一任务产出相同的分类。这很重要，因为一旦两个工具对同一变更的风险判断不一致，治理就失去了可信度——如果"Standard"在不同 Agent 里含义不同，团队就会停止信任分类，开始无视它。

The deeper takeaway: a one-dimensional risk model is a premature compression. It forces the classifier to make a judgment call that should be deferred to policy. Three independent axes let the harness policy decide what combinations matter — and that policy can evolve without changing the classification model. 更深层的收获：一维风险模型是过早的信息压缩。它把本该由策略层做的判断强加给了分类器。三个独立轴让 harness 策略来决定哪些组合重要——策略可以演进而不用改分类模型。

Confirmation Gates: The Async Blind Spot确认门控：异步盲区

Original harness: confirmation gates for Critical tasks only. Sound logic for the CLI agent — the user is watching the terminal, can interrupt mid-execution if something looks off. Standard tasks presented a plan and continued unless the user objected. In practice, CLI users objected within seconds when they disagreed. 原始 harness：只对 Critical 任务有确认门控。对 CLI Agent 来说逻辑没问题——用户盯着终端，执行过程中发现不对可以随时中断。Standard 任务展示计划后继续执行，除非用户反对。实际上 CLI 用户不同意时几秒钟内就会打断。

Then a user submitted "refactor the auth module" via the async agent and went to lunch. The agent classified it as Standard — business logic is L1 sensitivity — presented the plan in the task queue, got no objection (because no one was reading), and started rewriting three files. The user came back to find changes they never intended. Sound familiar? It's the same class of bug I described in the original harness article — the agent doesn't say "are you sure?" — except now the blast radius was bigger because the system assumed silence meant consent. 然后一个用户通过异步 Agent 提交了"重构认证模块"就去吃午饭了。Agent 分类为 Standard——业务逻辑是 L1 敏感度——在任务队列里展示了计划，没收到反对（因为没人在看），开始改三个文件。用户回来发现了一堆他们从没打算做的改动。听着耳熟？跟我在最初 harness 文章里描述的是同一类 bug——Agent 不会说"你确定吗？"——只是这次影响面更大，因为系统把沉默当成了同意。

The fix was declaring two execution modes in the spec — Interactive and Async — with confirmation semantics that vary by mode: 修复方法是在规范中声明两种执行模式——Interactive 和 Async——确认语义按模式变化：

  • Standard + Interactive: Present plan, proceed unless the user objects in the same turn. This is what the CLI already did.Standard + Interactive：展示计划，除非用户在同一轮反对否则继续。CLI 本来就是这样。
  • Standard + Async: Present plan, persist state, wait for explicit "confirmed" before touching any files.Standard + Async：展示计划，持久化状态，等到明确的"确认"再碰任何文件。
  • Critical (either mode): Always wait. No "execute all" shortcut — each step needs acknowledgment. The async handler flat-out rejects skip-checkpoint messages at Critical.Critical（无论哪种模式）：必须等待。没有"全部执行"的捷径——每个步骤都要确认。异步 Agent 在 Critical 级别直接拒绝跳过检查点的请求。

That last point — rejecting "just do it all" for Critical tasks — came from a lesson we'd already learned the hard way. In the earlier harness, we had a fuzzy confirmation matcher that triggered on substrings: if "好" in message. A user once wrote "不好意思我在忙" ("sorry, I'm busy") and the system treated the embedded "好" character as approval. After that incident, every approval gate uses exact-match whitelisting. When you're guarding against irreversible production changes, you err on the side of explicit, not convenient. 最后一点——对 Critical 任务拒绝"全部执行"——来自我们之前踩过的坑。早期 harness 用子字符串做模糊确认匹配：if "好" in message。有个用户写了"不好意思我在忙"，系统把里面的"好"字当成了批准。那次事件之后，所有审批门控都用精确匹配白名单。当你守着不可逆的生产变更时，宁可麻烦一点，也不要图方便。

Delivery Reports: Five Rounds to Get It Right交付报告：五轮才搞定

This part is less about systems design and more about me repeatedly getting the requirements wrong. I'm including the full iteration history because I think the pattern — starting from instinct, failing, then finally checking what established systems actually do — is more common than people admit. 这部分与其说是系统设计，不如说是我反复搞错需求。我把完整的迭代历史放在这里，因为我觉得这个模式——从直觉出发、失败、最后才去查成熟系统怎么做——比大多数人愿意承认的要常见得多。

Neither agent had a structured delivery report. The CLI agent had a loose table with risk level and acceptance results. The async agent had its own format with different fields. When we unified the harness, we needed a standard template both agents could produce. Seemed straightforward. 两个 Agent 都没有结构化交付报告。CLI Agent 有一个松散的表格包含风险等级和验收结果。异步 Agent 有自己的格式、不同的字段。统一 harness 时我们需要一个两个 Agent 都能产出的标准模板。看着挺简单。

Round 1 — I designed a table with six mandatory fields: risk level, change scope, execution steps, verification results, residual risk, unfinished items. Thorough on paper. In practice, a typo fix now generated a report with four "N/A" rows. The ceremony was absurd for the work. 第 1 轮——我设计了一个六个必填字段的表格：风险等级、变更范围、执行步骤、验证结果、残余风险、未完成项。纸面上很全面。实际运行中，一个拼写错误修复现在会生成四行"N/A"的报告。仪式感和工作量完全不成比例。

Round 2 — Feedback said "all workspace tasks should produce a fixed report." So I made all five sections mandatory. Empty? Fill in "none." Result: every small task produced a bloated report. The team started ignoring them entirely — the exact opposite of what we wanted. 第 2 轮——反馈说"workspace 里面的任务也要有固定的交付报告"。于是我让五个 section 全部必填。空的？填"无"。结果：每个小任务都产出臃肿的报告。团队开始完全无视它们——和我们想要的效果完全相反。

Round 3 — Pushed further: "Why isn't the template completely fixed?" So I made it rigid — same depth for typo fixes and database migrations. At this point I stopped and asked myself: does any established engineering practice actually recommend uniform-depth documentation regardless of change risk? 第 3 轮——继续推："交付报告为什么不完全固定？"于是搞成了完全刚性——修复拼写错误和数据库迁移用同样的深度。这时候我停下来问自己：到底有没有任何成熟的工程实践推荐不管变更风险大小都用统一深度的文档？

Round 4 — I should have done this from the start. I went and studied eight templates that have survived real-world usage at scale: 第 4 轮——我早该这么做。我去研究了八个经过大规模实际使用检验的模板：

  1. Claude CLI PR template (## Summary + ## Test plan)Claude CLI PR 模板（## Summary + ## Test plan）
  2. Google's CL description guide (what → why → how to verify)Google CL 描述指南（what → why → how to verify）
  3. React PR template (What? → Why? → How did you test?)React PR 模板（What? → Why? → How did you test?）
  4. Next.js PR template (description + test plan)Next.js PR 模板（description + test plan）
  5. Conventional Commits (type(scope): description)Conventional Commits（type(scope): description）
  6. ITIL Change Record (description → risk → verification → back-out plan)ITIL Change Record（description → risk → verification → back-out plan）
  7. DORA State of DevOps findings on documentation overhead vs. delivery speedDORA State of DevOps 关于文档开销与交付速度的研究

The pattern across all seven was unanimous: What → Why → How verified. Three questions. Depth scales with risk. Claude CLI wants two sections. Google wants three sentences. ITIL wants six fields with a back-out plan — but only for high-risk changes. Not one of these templates enforces the same depth for a typo fix and a database migration. DORA's research is particularly clear: heavyweight documentation ceremonies slow delivery without improving outcomes. Unread reports provide exactly zero auditability. 七个模板的共同模式非常一致：What → Why → How verified。三个问题。深度随风险缩放。Claude CLI 要两个 section。Google 要三句话。ITIL 要六个字段加回退计划——但只针对高风险变更。没有一个模板对拼写修复和数据库迁移强制相同深度。DORA 的研究说得特别清楚：重量级文档仪式拖慢交付而不改善结果。没人读的报告提供的可审计性为零。

Round 5 — the final design. Five sections, depth proportional to risk: 第 5 轮——最终设计。五个 section，深度与风险成比例：

  1. Verdict line — risk level + status + one-sentence conclusion. Always present. Borrowed from Conventional Commits: the subject line tells you whether to read further.结论行——风险等级 + 状态 + 一句话结论。始终存在。借鉴 Conventional Commits：主题行告诉你要不要继续读。
  2. Summary — why it was done, what happened. 1–3 bullets. Same as Google CL descriptions and React's What? Why? sections.Summary——为什么做、做了什么。1-3 个 bullet。和 Google CL 描述、React 的 What? Why? 一样。
  3. Changes — semantic description per file. Always present. Not just paths — "added token expiry check," not "modified auth.py."Changes——每个文件的语义描述。始终存在。不只是路径——"添加 token 过期校验"，而不是"修改了 auth.py"。
  4. Verification — how it was verified and what the results were. Standard and Critical only. Same question as Claude CLI's ## Test plan and React's How did you test?.Verification——怎么验证的、结果是什么。仅 Standard 和 Critical。和 Claude CLI 的 ## Test plan、React 的 How did you test? 是同一个问题。
  5. Issues — residual risk, unfinished items, rollback command. Critical must include rollback, same as ITIL's back-out plan. Omitted entirely if there's nothing to report.Issues——残余风险、未完成项、回滚命令。Critical 必须包含回滚，和 ITIL 的 back-out plan 一样。没有就不写。

A Routine typo fix produces three lines. A Critical database migration produces a full report with rollback instructions and residual risk. Same template, different depth. Nobody writes "N/A" four times for a one-line change anymore. 一个 Routine 级拼写修复产出三行。一个 Critical 级数据库迁移产出完整报告，包含回滚指令和残余风险。同一个模板，不同的深度。不再有人为了一行改动写四遍"N/A"了。

Looking back, rounds 1–3 were wasted. I was iterating on instinct — "make it thorough," "make it mandatory" — instead of checking what systems that handle millions of changes per year actually settled on. The answer was right there in the Google CL guide, which has governed every CL at Google for over a decade. Lesson learned: when you're designing a process template, check what survived at scale before designing from scratch. 回头看，第 1-3 轮是白费的。我在凭直觉迭代——"做全面点"、"做成强制的"——而没有去查每年处理数百万变更的系统最终选择了什么。答案就在 Google CL 指南里，那套东西管了 Google 十几年的每一个变更。教训：设计流程模板时，先看看大规模验证过的实践，再从零开始设计。

One Spec, Dual Modes一套规范，双模式

With risk, confirmation, and reporting aligned, we could actually merge. The key realization was simple in hindsight: Interactive and Async aren't different harnesses. They're different modes of the same harness. The gates — risk classification, scope confirmation, execution, validation, delivery, post-task capture — are identical. Only the confirmation mechanism varies. 风险、确认、报告对齐之后，终于可以合并了。事后看关键认知很简单：Interactive 和 Async 不是不同的 harness。它们是同一个 harness 的不同模式。门控——风险分类、范围确认、执行、验证、交付、任务后捕获——完全一致。只有确认机制不同。

Now one spec file governs both agents. The async orchestrator implements the Async mode of the shared harness — it doesn't maintain its own gate definitions. When the spec evolves, both agents evolve. We unified all internal nomenclature so there's one vocabulary across the codebase — no more parallel naming systems drifting apart. 现在一个规范文件治理两个 Agent。异步编排平台实现共享 harness 的 Async 模式——不再维护自己的门控定义。规范演进时两个 Agent 一起演进。我们统一了所有内部命名，整个代码库一套词汇表——不再有两套命名各自漂移。

The Memory Pipeline Fix记忆管道修复

Back to the silent data loss that started this whole effort. The async agent wrote execution records using episode_id, task_scope, task_outcome. The CLI agent's capture pipeline expected id, scope, outcome. The downstream aggregation — extracting insights, compacting old records, syncing to the memory system — only spoke the CLI's dialect. 40% of the async agent's data was being silently discarded. 回到引发这一切的静默数据丢失问题。异步 Agent 往 JSONL 写执行记录用的是 episode_id、task_scope、task_outcome。CLI Agent 的捕获管道期望的是 id、scope、outcome。下游聚合——提取洞察、压缩旧记录、同步到记忆系统——只认 CLI 的方言。异步 Agent 40% 的数据在被悄悄丢弃。

Instead of adding a translation layer (which just moves the schema problem), we made the async agent import the CLI's capture function directly. One shared function handles schema normalization, content sanitization, and file rotation. One code path, one schema, one downstream pipeline. The harness spec now documents alternative capture methods for agents that can't call CLI commands — import the function or write to the canonical schema directly, as long as you run the sanitization step. 我们没有加翻译层（那只是把 schema 问题换了个地方），而是让异步 Agent 直接导入 CLI 的捕获函数。一个共享函数负责 schema 规范化、内容脱敏和文件轮转。一条代码路径、一个 schema、一条下游管道。Harness 规范现在记录了替代捕获方式，面向无法调用 CLI 命令的 Agent——导入函数或直接写规范 schema，前提是走脱敏步骤。

What I'd Do Differently如果重来我会怎么做

Three mistakes, in order of cost: 三个错误，按代价从大到小排：

Define the schema before building implementations. We let two agents build their own episode schemas for three months, then had to reconcile them. Schema-first would have caught episode_id vs id before any code was written. This is the same principle as API-first design, and I should have recognized it sooner — we enforce it for our external APIs but didn't think to apply it to internal agent pipelines. 先定义 schema 再建实现。我们让两个 Agent 各自建了三个月的 episode schema，然后才来调和。Schema-first 会在写代码之前就发现 episode_id 对 id 的问题。这和 API-first 设计是同一个原则，我应该更早意识到——我们对外部 API 强制执行这个原则，但没想到要用在内部 Agent 管道上。

Design for dual-mode from the start. The Interactive-only assumption was baked into every confirmation gate. Retrofitting Async mode meant touching all of them. If we'd declared both modes upfront, integrating the second agent would have been configuration, not redesign. 从一开始就设计双模式。仅 Interactive 的假设嵌入了每一个确认门控。改造 Async 模式意味着全部要改。如果一开始就声明两种模式，接入第二个 Agent 就是配置变更，不是重新设计。

Research before iterating. The delivery report could have been done in three rounds instead of five. Rounds 1–3 were spent iterating on gut feeling; round 4 was "let me actually check what works at scale." I lost two iterations to the assumption that I could design a good template from first principles. I couldn't. The people who run Google's CL process and ITIL's change management have more data points than I do. Should have started there. 先调研再迭代。交付报告本可以三轮搞定而不是五轮。第 1-3 轮在凭感觉迭代；第 4 轮才是"让我实际查一下大规模验证过的做法"。我浪费了两轮在"我能从第一性原理设计出好模板"这个假设上。结果不行。管 Google CL 流程和 ITIL 变更管理的人有比我多得多的数据点。应该从那里开始。

Where It Stands Now现在的状态

The unified harness has six gates — risk classification, scope confirmation, execution, validation, delivery, and post-task capture — with risk-proportional depth. Routine tasks (all L0) skip planning and formal validation. Standard tasks (any L1) get the full flow. Critical tasks (any L2) get everything plus mandatory human confirmation before execution begins. Both agents implement the same spec, produce the same reports, and feed the same execution record pipeline. 统一后的 harness 有六个门控——风险分类、范围确认、执行、验证、交付、任务后捕获——深度与风险成比例。Routine 任务（全 L0）跳过规划和正式验证。Standard 任务（任一 L1）走完整流程。Critical 任务（任一 L2）全套加上执行前的强制人工确认。两个 Agent 实现同一规范，产出同样的报告，输入同一条执行记录管道。

If there's one line I'd keep from this whole exercise, it's this: a spec with only one implementation isn't a spec — it's a description of that implementation. You find out what's actually load-bearing when a second system tries to follow the same rules. We didn't design the three-axis risk model or the dual-mode confirmation or the proportional reporting because we had foresight. We designed them because a second agent showed up and the original spec couldn't handle it. 如果让我从这整个过程中只留一句话：只有一个实现的规范不是规范——它是对那个实现的描述。当第二个系统尝试遵循同一套规则时，你才发现什么是真正承重的。我们不是因为有远见才设计了三轴风险模型、双模式确认和比例化报告。是因为第二个 Agent 出现了，原始规范扛不住了。

Checklist: Governing Multiple Agents清单：多 Agent 治理

If you're about to add a second agent to a governance spec that currently only has one, here's what I'd check first: 如果你正要把第二个 Agent 接入一套目前只有一个实现的治理规范，以下是我建议先检查的：

  1. Define execution record schema before building implementations. Field names, required fields, enum values. If two agents invent their own schemas for three months, you'll spend a week reconciling them — or worse, lose data silently.先定义执行记录的 schema 再写实现。字段名、必填项、枚举值。如果两个 Agent 各自发明 schema 三个月，你会花一周来调和——或者更糟，静默丢数据。
  2. Use independent risk axes, not a single dimension. Blast radius, reversibility, and sensitivity each contribute different signals. Let policy compose them (take the max), don't force the classifier to compress them into one label.用独立的风险轴，不要用单一维度。影响面、可逆性、敏感度各自贡献不同信号。让策略层来组合（取最大值），不要强迫分类器把它们压成一个标签。
  3. Declare Interactive vs Async modes explicitly. If a user might not be present when the agent acts, silence cannot mean consent. Async mode must checkpoint and wait.显式声明 Interactive 和 Async 模式。如果 Agent 执行时用户可能不在，沉默不能等于同意。Async 模式必须设检查点并等待。
  4. Scale report depth with risk. What → Why → How verified. Three questions, not six mandatory fields. Routine gets a one-liner. Critical gets rollback instructions. Check Google's CL guide, React's PR template, and ITIL before designing from scratch.报告深度随风险缩放。What → Why → How verified。三个问题，不是六个必填字段。Routine 一句话。Critical 要回滚指令。设计之前先查 Google CL 指南、React PR 模板和 ITIL。
  5. One capture path, one schema, one pipeline. Don't add a translation layer between agent-specific formats — that just moves the schema problem. Make every agent write to the same canonical format.一条捕获路径、一个 schema、一条管道。不要在 Agent 各自的格式之间加翻译层——那只是把 schema 问题换个地方。让每个 Agent 都写同一个规范格式。
  6. Test your spec with two implementations. One implementation proves the spec compiles. Two prove it's meaningful. If you only have one agent, your spec is a description of that agent — you won't know what's actually load-bearing until the second one shows up.用两个实现验证你的规范。一个实现证明规范能跑通。两个实现证明规范有意义。如果你只有一个 Agent，你的规范就是对那个 Agent 的描述——第二个出现之前你不会知道什么是真正承重的。
