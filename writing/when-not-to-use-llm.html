When Not to Use an LLM 什么时候不该用 LLM

If you're reaching for an LLM every time you need to transform data, classify a request, or validate input, stop and ask yourself: does this actually need language understanding? Every LLM call in your pipeline adds latency, cost, and a non-determinism surface. The default should be skepticism — you need a clear reason to add an LLM call, not a clear reason to avoid it. This article gives you a five-question framework and two real-world cases to help you draw that line. 如果你每次需要转换数据、分类请求或验证输入时都想用 LLM，请先停下来问自己：这真的需要语言理解吗？管道中的每个 LLM 调用都会增加延迟、成本和不确定性。默认态度应该是怀疑——你需要一个明确的理由来添加 LLM 调用，而不是一个明确的理由来避免它。本文提供一个五问框架和两个真实案例，帮你划清这条线。

Key Takeaways核心要点

  • Every LLM call adds four compounding costs — latency, non-determinism, compute expense, and audit complexity — that must be justified on every single request.每次 LLM 调用都会叠加四项成本——延迟、不确定性、计算费用和审计复杂性——必须在每个请求上逐一论证其合理性。
  • Replacing 9 out of 23 LLM calls with deterministic code cut monthly inference cost by 59% and reduced non-determinism incidents from 7 per month to 1.将 23 个 LLM 调用中的 9 个替换为确定性代码，月推理成本降低了 59%，不确定性事故从每月 7 起降至 1 起。
  • A keyword trie plus lightweight TF-IDF matched LLM routing accuracy at 97.3% while resolving in under 2ms versus the LLM's 780ms.关键词 trie 加轻量级 TF-IDF 以 97.3% 的准确率匹配了 LLM 路由结果，解析时间不到 2ms，而 LLM 需要 780ms。
  • Using an LLM to patch messy upstream data creates two layers of uncertainty — fix the data pipeline instead of hiding the problem behind inference.使用 LLM 来修补混乱的上游数据会制造两层不确定性——应该修复数据管道，而不是用推理掩盖问题。
  • The most reliable architecture lets the LLM handle understanding and generation while deterministic code enforces authorization, validation, and control.最可靠的架构让 LLM 负责理解和生成，而确定性代码负责授权、验证和控制。

The LLM TaxLLM 税

Every LLM call in your pipeline adds four costs that you have to pay on every single request: 管道中的每次 LLM 调用都增加了四项你必须在每个请求上支付的成本：

  • Latency. Even with optimized serving (vLLM, TensorRT-LLM), an LLM call adds hundreds of milliseconds to seconds. For synchronous pipelines, this directly impacts user experience.延迟。即使使用优化的推理服务（vLLM、TensorRT-LLM），一次 LLM 调用也会增加数百毫秒到秒级的延迟。对于同步管道，这直接影响用户体验。
  • Non-determinism. Same input, different output. This makes testing harder, debugging harder, and compliance harder. You can set temperature to 0, but you still don't get guaranteed reproducibility across model versions.不确定性。相同输入，不同输出。这使测试更难、调试更难、合规更难。你可以将 temperature 设为 0，但跨模型版本仍无法保证可复现性。
  • Compute cost. LLM inference is expensive. At scale, the difference between "use an LLM" and "use a regex" for a classification task can be orders of magnitude in infrastructure cost.计算成本。LLM 推理很昂贵。在规模化时，对于分类任务使用 LLM 还是使用正则表达式，基础设施成本可能相差几个数量级。
  • Audit complexity. In regulated environments, you need to explain why the system produced a specific output. Deterministic code gives you a complete, reproducible trace. An LLM gives you "it seemed right based on the weights."审计复杂性。在受监管环境中，你需要解释系统为什么产生特定输出。确定性代码给你一个完整、可复现的 trace。LLM 给你的是"基于权重看起来是对的"。

These costs are worth paying when the LLM provides something you genuinely can't get from deterministic code. They're not worth paying when you're using an LLM as a fancy if-else. 当 LLM 提供了你从确定性代码中真正无法获得的东西时，这些成本值得支付。当你把 LLM 当作花哨的 if-else 使用时，这些成本不值得。

Don't Use an LLM When...不该用 LLM 的场景...

The Task Is Deterministic任务是确定性的

If the mapping from input to output is fixed and known, write a function. Lookups, format conversions, field extractions from structured data, rule-based classifications — these don't need language understanding. A dictionary lookup is faster, cheaper, and guaranteed correct. 如果输入到输出的映射是固定的且已知的，就写一个函数。查找、格式转换、从结构化数据中提取字段、基于规则的分类——这些不需要语言理解。字典查找更快、更便宜，且保证正确。

I've seen teams use an LLM to classify tickets into 5 categories, when 90% of the classification could be done with keyword matching and the remaining 10% with a lightweight classifier. The LLM added 800ms of latency to every request for marginal accuracy improvement. 我见过团队使用 LLM 将工单分类为 5 个类别，而 90% 的分类可以通过关键词匹配完成，剩余 10% 使用轻量级分类器。LLM 为每个请求增加了 800ms 的延迟，却只换来边际准确性提升。

You Need Exact Reproducibility需要精确可复现性

If the same input must always produce the same output — for legal, regulatory, or contractual reasons — an LLM is the wrong tool. Deterministic code gives you this guarantee. LLMs don't, even at temperature 0, because model updates, quantization changes, and infrastructure differences can all affect output. 如果出于法律、监管或合同原因，相同的输入必须始终产生相同的输出——LLM 是错误的工具。确定性代码给你这个保证。LLM 不能，即使 temperature 为 0，因为模型更新、量化变化和基础设施差异都可能影响输出。

You're Papering Over Bad Data在掩盖糟糕的数据

A surprisingly common pattern: the upstream data is messy, so someone adds an LLM to "interpret" it. The LLM becomes a translation layer between bad data and the system that needs clean data. This works until it doesn't — and when it fails, debugging is nearly impossible because you're troubleshooting through two layers of uncertainty (the data and the LLM). 一个令人惊讶的常见模式：上游数据很混乱，所以有人添加一个 LLM 来"解释"它。LLM 变成了混乱数据和需要干净数据的系统之间的翻译层。这在它失效之前一直有效——而当它失败时，调试几乎不可能，因为你在排查两层不确定性（数据和 LLM）。

Fix the data pipeline. Write a proper parser. Add validation at the source. These are boring solutions, but they're debuggable. 修复数据管道。写一个正确的解析器。在源头添加验证。这些是无聊的解决方案，但它们是可调试的。

Latency Matters More Than Flexibility延迟比灵活性更重要

When the data schema is known and the query patterns are predictable, a SQL query will always be faster than RAG (Retrieval-Augmented Generation — feeding retrieved documents into an LLM to produce an answer). If your "AI-powered search" is really just querying a well-structured database, you're paying the LLM tax for something that SQL handles in milliseconds. 当数据 schema 已知且查询模式可预测时，SQL 查询总是比 RAG（检索增强生成——将检索到的文档输入 LLM 来生成答案）更快。如果你的"AI 驱动的搜索"实际上只是查询一个结构良好的数据库，你就是在为 SQL 毫秒级可以处理的事情支付 LLM 税。

Two Times I Almost Reached for an LLM两次我差点用了 LLM

These aren't hypotheticals. These are decisions I made in the last year that saved real money and real debugging hours. 这些不是假设。这些是我去年做出的决策，节省了真金白银和真实的调试时间。

The validation layer. We needed to validate incoming webhook payloads from 14 partner integrations — each with slightly different schemas, inconsistent field naming, and occasionally malformed JSON. My first instinct was to throw the payload at GPT-4 with a prompt like "extract and normalize these fields." I built a prototype in an afternoon and it worked well enough in testing. Then I benchmarked it: 620ms average per call, $0.003 per validation, and a 2.1% hallucination rate where the model would "fill in" missing fields with plausible but fabricated values. For a system processing 400K webhooks/day, that meant ~8,400 silently corrupted records daily. I replaced it with a schema registry plus a rule engine — 47 JSONPath rules (a query language for extracting values from JSON documents, similar to XPath for XML) and a handful of regex normalizers. Processing time dropped to 3ms per payload. Cost went to effectively zero. Hallucination rate: zero, by definition. The whole rewrite took four days. The LLM prototype took one afternoon but would have cost us $36K/year in API fees alone, not counting the data quality incidents. 验证层。我们需要验证来自 14 个合作伙伴集成的传入 webhook 负载——每个都有略微不同的 schema、不一致的字段命名，偶尔还有格式错误的 JSON。我的第一直觉是把负载扔给 GPT-4，配上"提取并标准化这些字段"的提示。我用一个下午构建了原型，在测试中表现不错。然后我做了基准测试：平均每次调用 620ms，每次验证 $0.003，还有 2.1% 的幻觉率——模型会用看似合理但虚假的值"填充"缺失字段。对于一个每天处理 40 万个 webhook 的系统，这意味着每天约 8,400 条静默损坏的记录。我用 schema 注册表加规则引擎替代了它——47 条 JSONPath 规则（一种从 JSON 文档中提取值的查询语言，类似于 XML 的 XPath）和几个正则标准化器。处理时间降到每个负载 3ms。成本实际上为零。幻觉率：从定义上就是零。整个重写花了四天。LLM 原型花了一个下午，但仅 API 费用每年就要 3.6 万美元，还不算数据质量事故。

The routing decision. Our agent platform needed to route incoming user requests to one of 12 specialized sub-agents. The obvious approach was an LLM classifier — feed it the user message, get back a category. It worked, but added 780ms to every single interaction before the actual agent even started processing. For a system where perceived responsiveness matters enormously, that was unacceptable. I spent a week analyzing 50K historical requests and found that a keyword-based trie (a tree structure that matches strings by prefix, enabling O(k) lookup where k is the key length) covering ~200 terms, combined with a lightweight TF-IDF model (Term Frequency-Inverse Document Frequency — a classic statistical method that scores how important a word is to a document relative to a corpus) for the ambiguous cases (about 15% of traffic), matched the LLM's routing accuracy at 97.3% while resolving in under 2ms. The remaining 2.7% of "misroutes" were cases where even the LLM was inconsistent across runs. We shipped the trie. Median first-token latency for user-facing responses dropped from 1.4s to 0.6s overnight. 路由决策。我们的 Agent 平台需要将传入的用户请求路由到 12 个专门的子 Agent 之一。显而易见的方法是 LLM 分类器——把用户消息喂给它，返回一个类别。它能用，但在实际 Agent 开始处理之前，给每次交互增加了 780ms。对于一个感知响应速度极其重要的系统来说，这是不可接受的。我花了一周分析了 5 万条历史请求，发现一个覆盖约 200 个术语的基于关键词的 trie（一种按前缀匹配字符串的树结构，查找时间为 O(k)，k 为键长），结合一个轻量级 TF-IDF 模型（词频-逆文档频率——一种经典统计方法，衡量词语相对于语料库的重要性）处理模糊情况（大约 15% 的流量），在 2ms 内解决，匹配了 LLM 97.3% 的路由准确率。剩余 2.7% 的"误路由"是即使 LLM 在不同运行中也不一致的情况。我们上线了 trie。面向用户的首 token 中位延迟一夜之间从 1.4 秒降到了 0.6 秒。

A Decision Framework决策框架

Before adding an LLM call to your pipeline, answer five questions: 在向管道添加 LLM 调用之前，回答五个问题：

  1. Is the input structured or unstructured? If structured, you probably don't need an LLM to understand it.输入是结构化的还是非结构化的？如果是结构化的，你可能不需要 LLM 来理解它。
  2. Is the output format fixed or flexible? Fixed-format outputs mean the LLM is just a formatter. Jinja2 is faster and doesn't hallucinate field names.输出格式是固定的还是灵活的？固定格式的输出意味着 LLM 只是一个格式化器。Jinja2 更快，而且不会幻觉出字段名。
  3. Is there a human in the loop? Without one, every LLM error goes straight to production. That changes the risk calculus entirely.是否有人在环路中？如果没有，每个 LLM 错误都直接进入生产。这完全改变了风险计算。
  4. What's the cost of a wrong answer? In financial and legal contexts, we gate every LLM output through deterministic checks and human sign-off. No exceptions.错误答案的代价是什么？在财务和法律场景中，我们对每一个 LLM 输出都通过确定性检查和人工签核进行门控。没有例外。
  5. How do you evaluate correctness? If you can't define a metric for "correct," you can't tell if the LLM is helping or hurting.如何评估正确性？如果你无法定义"正确"的指标，你就无法判断 LLM 是在帮忙还是在添乱。

If you answer "structured, fixed, no human, high cost, hard to evaluate," that's five signals pointing away from an LLM. 如果你的回答是"结构化、固定、无人、高代价、难以评估"，那就是五个远离 LLM 的信号。

The Right Boundary: LLM Decides What, Rules Decide Whether正确的边界：LLM 决定"做什么"，规则决定"是否做"

The most effective pattern I've found is a clear separation: the LLM handles understanding and generation (what to say, what to do), while deterministic code handles authorization and control (whether to say it, whether to do it). 我发现最有效的模式是清晰的分离：LLM 处理理解和生成（说什么、做什么），确定性代码处理授权和控制（是否说、是否做）。

In our agent platform: 在我们的 Agent 平台中：

  • The LLM generates a candidate answer — but deterministic gating validates it against evidence thresholds, jurisdiction constraints, and entitlements before delivery.LLM 生成候选答案——但确定性门控在交付前根据证据阈值、管辖区约束和权限验证它。
  • The LLM plans a multi-step workflow — but the execution engine enforces step budgets, circuit breakers, and least-privilege constraints deterministically.LLM 规划多步骤工作流——但执行引擎以确定性方式执行步骤预算、熔断器和最小权限约束。
  • The LLM drafts a review package — but the maker-checker framework requires human sign-off through codified, auditable rules.LLM 起草审查包——但 maker-checker 框架通过编码化、可审计的规则要求人工签核。

This isn't a compromise — it's a design principle. The LLM handles understanding and generation — natural language, synthesis, structured content. Everything else stays in deterministic code, where it's fast, testable, and auditable. We enforce this split at the architecture level, not through developer discipline. 这不是妥协——这是一个设计原则。LLM 做它独特擅长的事——理解自然语言、综合信息、生成结构化内容。其他一切都留在确定性代码中，在那里它快速、可测试、可审计。我们在架构层面强制执行这一分工，而非依赖开发者自律。

Last quarter I audited our platform's inference spend. Of 23 LLM calls in the pipeline, we eliminated 9 by replacing them with deterministic code. Monthly inference cost dropped from $14,200 to $5,800. Median end-to-end latency fell by 40%. The number of production incidents tagged "non-deterministic behavior" went from 7 per month to 1. The one thing I'm still not sure about: where the line sits for tasks that are 80% deterministic and 20% genuinely ambiguous. We currently run those through the LLM, but I suspect there's a hybrid approach — deterministic fast path with LLM fallback — that we haven't built yet. If you've figured that pattern out, I'd like to hear about it. 上个季度我审计了平台的推理开支。管道中 23 个 LLM 调用里，我们通过替换为确定性代码消除了 9 个。月推理成本从 14,200 美元降到 5,800 美元。中位端到端延迟下降了 40%。标记为"不确定性行为"的生产事故从每月 7 起降到 1 起。我仍然不确定的一件事是：对于 80% 确定性、20% 真正模糊的任务，界限在哪里。我们目前把这些交给 LLM 处理，但我怀疑有一种混合方法——确定性快速路径加 LLM 回退——我们还没构建出来。如果你已经摸索出了这个模式，我很想听听。

Checklist: Before You Add an LLM Call检查清单：在添加 LLM 调用之前

Use this checklist every time you consider adding an LLM call to a pipeline. If you check three or more boxes, you probably don't need one. 每次考虑向管道添加 LLM 调用时，请使用此检查清单。如果你勾选了三项或更多，你很可能不需要 LLM。

  • Input is structured — The data has a known schema. A parser or query can extract what you need without language understanding.输入是结构化的——数据有已知的 schema。解析器或查询就能提取你需要的内容，不需要语言理解。
  • Output format is fixed — The response shape never changes. A template engine (Jinja2, Handlebars) is faster and deterministic.输出格式是固定的——响应形状永远不变。模板引擎（Jinja2、Handlebars）更快且确定性。
  • No human in the loop — Every error goes straight to production. Consider whether you can tolerate the LLM's hallucination rate at your request volume.没有人在环路中——每个错误都直接进入生产。考虑在你的请求量下能否容忍 LLM 的幻觉率。
  • Wrong answers are expensive — Financial, legal, or safety consequences. Gate every LLM output through deterministic validation and human sign-off.错误答案代价高昂——有财务、法律或安全后果。对每个 LLM 输出都通过确定性验证和人工签核进行门控。
  • Correctness is hard to measure — If you can't define a metric for "correct," you can't tell whether the LLM is helping or hurting.正确性难以衡量——如果你无法定义"正确"的指标，你就无法判断 LLM 是在帮忙还是在添乱。
  • Latency budget is tight — Your p95 latency target leaves no room for the 200ms-2s overhead of an LLM call. Benchmark before committing.延迟预算紧张——你的 p95 延迟目标没有为 LLM 调用的 200ms-2s 开销留下空间。先做基准测试再决定。
  • A deterministic alternative exists — Regex, rule engine, lookup table, lightweight classifier. If one of these gets you to 95%+ accuracy, start there and add an LLM fallback only for the long tail.存在确定性替代方案——正则表达式、规则引擎、查找表、轻量级分类器。如果其中之一能达到 95%+ 准确率，先从它开始，仅对长尾情况添加 LLM 回退。

When you do use an LLM, enforce the boundary: LLM decides what, deterministic code decides whether. Let the LLM handle understanding and generation. Let rules, validators, and humans handle authorization and control. Enforce this split at the architecture level, not through developer discipline. 当你确实使用 LLM 时，强制执行边界：LLM 决定"做什么"，确定性代码决定"是否做"。让 LLM 处理理解和生成。让规则、验证器和人类处理授权和控制。在架构层面强制执行此分工，而非依赖开发者自律。

Frequently Asked Questions常见问题

Can you use an LLM as a fallback for edge cases in an otherwise deterministic pipeline?可以在确定性管道中将 LLM 作为边缘情况的回退吗？

Yes, and this is often the best hybrid pattern. Route the 85-95% of traffic you can handle deterministically through rules or classifiers, and send only the ambiguous remainder to an LLM. This caps your inference cost at a fraction of full-LLM throughput while preserving quality on the long tail. The key is building a confidence score on the deterministic path so you know exactly when to escalate.可以，这通常是最佳的混合模式。将 85-95% 可以确定性处理的流量通过规则或分类器路由，仅将模糊的剩余部分发送给 LLM。这将推理成本控制在全 LLM 吞吐量的一小部分，同时保持长尾质量。关键是在确定性路径上构建置信度分数，以便准确知道何时升级。

How do you benchmark an LLM call against a deterministic alternative before committing to either?在决定采用哪种方案之前，如何对 LLM 调用和确定性替代方案进行基准测试？

Build both paths as thin prototypes and measure four metrics on the same sample of real traffic: p50/p95 latency, accuracy against a human-labeled ground truth set, per-request cost, and failure mode distribution. The ground truth set is critical — without it you are comparing vibes, not performance. A 500-example labeled set is usually enough to make the decision.将两条路径都构建为轻量原型，并在同一批真实流量样本上测量四个指标：p50/p95 延迟、与人工标注的真值集对比的准确率、每请求成本和故障模式分布。真值集是关键——没有它，你比较的是感觉而非性能。通常 500 个标注示例就足以做出决策。

Does the five-question framework apply differently for internal tools versus customer-facing products?五问框架对内部工具和面向客户的产品适用方式不同吗？

The questions are the same, but the thresholds shift. Internal tools can often tolerate higher latency and occasional errors because there is a human operator who can catch and correct mistakes. Customer-facing products have tighter latency budgets and no safety net, so the bar for adding an LLM call is higher. The "human in the loop" question effectively carries double weight in customer-facing contexts.问题是一样的，但阈值不同。内部工具通常可以容忍更高的延迟和偶尔的错误，因为有人类操作员可以发现并纠正错误。面向客户的产品有更紧的延迟预算且没有安全网，因此添加 LLM 调用的门槛更高。"是否有人在环路中"这个问题在面向客户的场景中实际上权重翻倍。

What happens when a model provider updates the model version behind your API key?当模型提供商更新你 API 密钥背后的模型版本时会发生什么？

Silent model updates are one of the most underappreciated risks of LLM dependencies. Output distributions shift, edge-case behavior changes, and your carefully tuned prompts may degrade — all without any deployment on your side. Pin to a specific model version where your provider allows it, maintain a regression test suite of at least 200 representative inputs, and run it on a schedule. Treat model version changes like dependency upgrades: test before you ship.静默模型更新是 LLM 依赖中最被低估的风险之一。输出分布变化、边缘情况行为改变，你精心调优的提示可能退化——而你这边没有任何部署。在提供商允许的情况下锁定特定模型版本，维护至少 200 个代表性输入的回归测试套件，并定期运行。将模型版本变更视为依赖升级：上线前先测试。
