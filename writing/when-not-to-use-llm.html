When Not to Use an LLM 什么时候不该用 LLM

After two years of building production AI agent systems, the most valuable skill I've developed isn't prompt engineering or RAG optimization. It's knowing when to reach for an LLM and when to write a regular function. The default should be skepticism — every LLM call in a pipeline is latency, cost, and a non-determinism surface. You need a clear reason to add one, not a clear reason to avoid it. 在构建生产级 AI Agent 系统两年后，我培养的最有价值的技能不是提示工程或 RAG 优化。而是知道何时使用 LLM、何时写一个普通函数。我认识的最好的 AI 工程师都有这个特质：他们对在任何管道中添加 LLM 推理深表怀疑，并且有一个清晰的框架来判断何时值得付出代价。

The LLM TaxLLM 税

Every LLM call in your pipeline adds four costs that you have to pay on every single request: 管道中的每次 LLM 调用都增加了四项你必须在每个请求上支付的成本：

  • Latency. Even with optimized serving (vLLM, TensorRT-LLM), an LLM call adds hundreds of milliseconds to seconds. For synchronous pipelines, this directly impacts user experience.延迟。即使使用优化的推理服务（vLLM、TensorRT-LLM），一次 LLM 调用也会增加数百毫秒到秒级的延迟。对于同步管道，这直接影响用户体验。
  • Non-determinism. Same input, different output. This makes testing harder, debugging harder, and compliance harder. You can set temperature to 0, but you still don't get guaranteed reproducibility across model versions.不确定性。相同输入，不同输出。这使测试更难、调试更难、合规更难。你可以将 temperature 设为 0，但跨模型版本仍无法保证可复现性。
  • Compute cost. LLM inference is expensive. At scale, the difference between "use an LLM" and "use a regex" for a classification task can be orders of magnitude in infrastructure cost.计算成本。LLM 推理很昂贵。在规模化时，对于分类任务使用 LLM 还是使用正则表达式，基础设施成本可能相差几个数量级。
  • Audit complexity. In regulated environments, you need to explain why the system produced a specific output. Deterministic code gives you a complete, reproducible trace. An LLM gives you "it seemed right based on the weights."审计复杂性。在受监管环境中，你需要解释系统为什么产生特定输出。确定性代码给你一个完整、可复现的 trace。LLM 给你的是"基于权重看起来是对的"。

These costs are worth paying when the LLM provides something you genuinely can't get from deterministic code. They're not worth paying when you're using an LLM as a fancy if-else. 当 LLM 提供了你从确定性代码中真正无法获得的东西时，这些成本值得支付。当你把 LLM 当作花哨的 if-else 使用时，这些成本不值得。

Don't Use an LLM When...不该用 LLM 的场景...

The Task Is Deterministic任务是确定性的

If the mapping from input to output is fixed and known, write a function. Lookups, format conversions, field extractions from structured data, rule-based classifications — these don't need language understanding. A dictionary lookup is faster, cheaper, and guaranteed correct. 如果输入到输出的映射是固定的且已知的，就写一个函数。查找、格式转换、从结构化数据中提取字段、基于规则的分类——这些不需要语言理解。字典查找更快、更便宜，且保证正确。

I've seen teams use an LLM to classify tickets into 5 categories, when 90% of the classification could be done with keyword matching and the remaining 10% with a lightweight classifier. The LLM added 800ms of latency to every request for marginal accuracy improvement. 我见过团队使用 LLM 将工单分类为 5 个类别，而 90% 的分类可以通过关键词匹配完成，剩余 10% 使用轻量级分类器。LLM 为每个请求增加了 800ms 的延迟，却只换来边际准确性提升。

You Need Exact Reproducibility需要精确可复现性

If the same input must always produce the same output — for legal, regulatory, or contractual reasons — an LLM is the wrong tool. Deterministic code gives you this guarantee. LLMs don't, even at temperature 0, because model updates, quantization changes, and infrastructure differences can all affect output. 如果出于法律、监管或合同原因，相同的输入必须始终产生相同的输出——LLM 是错误的工具。确定性代码给你这个保证。LLM 不能，即使 temperature 为 0，因为模型更新、量化变化和基础设施差异都可能影响输出。

You're Papering Over Bad Data在掩盖糟糕的数据

A surprisingly common pattern: the upstream data is messy, so someone adds an LLM to "interpret" it. The LLM becomes a translation layer between bad data and the system that needs clean data. This works until it doesn't — and when it fails, debugging is nearly impossible because you're troubleshooting through two layers of uncertainty (the data and the LLM). 一个令人惊讶的常见模式：上游数据很混乱，所以有人添加一个 LLM 来"解释"它。LLM 变成了混乱数据和需要干净数据的系统之间的翻译层。这在它失效之前一直有效——而当它失败时，调试几乎不可能，因为你在排查两层不确定性（数据和 LLM）。

Fix the data pipeline. Write a proper parser. Add validation at the source. These are boring solutions, but they're debuggable. 修复数据管道。写一个正确的解析器。在源头添加验证。这些是无聊的解决方案，但它们是可调试的。

Latency Matters More Than Flexibility延迟比灵活性更重要

When the data schema is known and the query patterns are predictable, a SQL query will always be faster than RAG. If your "AI-powered search" is really just querying a well-structured database, you're paying the LLM tax for something that SQL handles in milliseconds. 当数据 schema 已知且查询模式可预测时，SQL 查询总是比 RAG 更快。如果你的"AI 驱动的搜索"实际上只是查询一个结构良好的数据库，你就是在为 SQL 毫秒级可以处理的事情支付 LLM 税。

Two Times I Almost Reached for an LLM两次我差点用了 LLM

These aren't hypotheticals. These are decisions I made in the last year that saved real money and real debugging hours. 这些不是假设。这些是我去年做出的决策，节省了真金白银和真实的调试时间。

The validation layer. We needed to validate incoming webhook payloads from 14 partner integrations — each with slightly different schemas, inconsistent field naming, and occasionally malformed JSON. My first instinct was to throw the payload at GPT-4 with a prompt like "extract and normalize these fields." I built a prototype in an afternoon and it worked well enough in testing. Then I benchmarked it: 620ms average per call, $0.003 per validation, and a 2.1% hallucination rate where the model would "fill in" missing fields with plausible but fabricated values. For a system processing 400K webhooks/day, that meant ~8,400 silently corrupted records daily. I replaced it with a schema registry plus a rule engine — 47 JSONPath rules and a handful of regex normalizers. Processing time dropped to 3ms per payload. Cost went to effectively zero. Hallucination rate: zero, by definition. The whole rewrite took four days. The LLM prototype took one afternoon but would have cost us $36K/year in API fees alone, not counting the data quality incidents. 验证层。我们需要验证来自 14 个合作伙伴集成的传入 webhook 负载——每个都有略微不同的 schema、不一致的字段命名，偶尔还有格式错误的 JSON。我的第一直觉是把负载扔给 GPT-4，配上"提取并标准化这些字段"的提示。我用一个下午构建了原型，在测试中表现不错。然后我做了基准测试：平均每次调用 620ms，每次验证 $0.003，还有 2.1% 的幻觉率——模型会用看似合理但虚假的值"填充"缺失字段。对于一个每天处理 40 万个 webhook 的系统，这意味着每天约 8,400 条静默损坏的记录。我用 schema 注册表加规则引擎替代了它——47 条 JSONPath 规则和几个正则标准化器。处理时间降到每个负载 3ms。成本实际上为零。幻觉率：从定义上就是零。整个重写花了四天。LLM 原型花了一个下午，但仅 API 费用每年就要 3.6 万美元，还不算数据质量事故。

The routing decision. Our agent platform needed to route incoming user requests to one of 12 specialized sub-agents. The obvious approach was an LLM classifier — feed it the user message, get back a category. It worked, but added 780ms to every single interaction before the actual agent even started processing. For a system where perceived responsiveness matters enormously, that was unacceptable. I spent a week analyzing 50K historical requests and found that a keyword-based trie covering ~200 terms, combined with a lightweight TF-IDF model for the ambiguous cases (about 15% of traffic), matched the LLM's routing accuracy at 97.3% while resolving in under 2ms. The remaining 2.7% of "misroutes" were cases where even the LLM was inconsistent across runs. We shipped the trie. Median first-token latency for user-facing responses dropped from 1.4s to 0.6s overnight. 路由决策。我们的 Agent 平台需要将传入的用户请求路由到 12 个专门的子 Agent 之一。显而易见的方法是 LLM 分类器——把用户消息喂给它，返回一个类别。它能用，但在实际 Agent 开始处理之前，给每次交互增加了 780ms。对于一个感知响应速度极其重要的系统来说，这是不可接受的。我花了一周分析了 5 万条历史请求，发现一个覆盖约 200 个术语的基于关键词的 trie，结合一个轻量级 TF-IDF 模型处理模糊情况（大约 15% 的流量），在 2ms 内解决，匹配了 LLM 97.3% 的路由准确率。剩余 2.7% 的"误路由"是即使 LLM 在不同运行中也不一致的情况。我们上线了 trie。面向用户的首 token 中位延迟一夜之间从 1.4 秒降到了 0.6 秒。

A Decision Framework决策框架

Before adding an LLM call to your pipeline, answer five questions: 在向管道添加 LLM 调用之前，回答五个问题：

  1. Is the input structured or unstructured? If structured, you probably don't need an LLM to understand it.输入是结构化的还是非结构化的？如果是结构化的，你可能不需要 LLM 来理解它。
  2. Is the output format fixed or flexible? Fixed-format outputs mean the LLM is just a formatter. Jinja2 is faster and doesn't hallucinate field names.输出格式是固定的还是灵活的？如果是固定的，LLM 只是一个格式化器——有更便宜的格式化方式。
  3. Is there a human in the loop? Without one, every LLM error goes straight to production. That changes the risk calculus entirely.是否有人在环路中？如果没有，LLM 错误的代价更高——你需要比 LLM 能提供的更强的保证。
  4. What's the cost of a wrong answer? In financial and legal contexts, we gate every LLM output through deterministic checks and human sign-off. No exceptions.错误答案的代价是什么？如果很高（财务、法律、安全），门控和人工审查必须包围任何 LLM 使用。
  5. How do you evaluate correctness? If you can't define a metric for "correct," you can't tell if the LLM is helping or hurting.如何评估正确性？如果你无法定义"正确"的指标，你就无法判断 LLM 是在帮忙还是在添乱。

If you answer "structured, fixed, no human, high cost, hard to evaluate," that's five signals pointing away from an LLM. 如果你的回答是"结构化、固定、无人、高代价、难以评估"，那就是五个远离 LLM 的信号。

The Right Boundary: LLM Decides What, Rules Decide Whether正确的边界：LLM 决定"做什么"，规则决定"是否做"

The most effective pattern I've found is a clear separation: the LLM handles understanding and generation (what to say, what to do), while deterministic code handles authorization and control (whether to say it, whether to do it). 我发现最有效的模式是清晰的分离：LLM 处理理解和生成（说什么、做什么），确定性代码处理授权和控制（是否说、是否做）。

In our agent platform: 在我们的 Agent 平台中：

  • The LLM generates a candidate answer — but deterministic gating validates it against evidence thresholds, jurisdiction constraints, and entitlements before delivery.LLM 生成候选答案——但确定性门控在交付前根据证据阈值、管辖区约束和权限验证它。
  • The LLM plans a multi-step workflow — but the execution engine enforces step budgets, circuit breakers, and least-privilege constraints deterministically.LLM 规划多步骤工作流——但执行引擎以确定性方式执行步骤预算、熔断器和最小权限约束。
  • The LLM drafts a review package — but the maker-checker framework requires human sign-off through codified, auditable rules.LLM 起草审查包——但 maker-checker 框架通过编码化、可审计的规则要求人工签核。

The LLM handles understanding and generation — natural language, synthesis, structured content. Everything else stays in deterministic code, where it's fast, testable, and auditable. We enforce this split at the architecture level, not through developer discipline. 这不是妥协。这是一个设计原则。LLM 做它独特擅长的事——理解自然语言、综合信息、生成结构化内容。其他一切都留在确定性代码中，在那里它快速、可测试、可审计。

Last quarter I audited our platform's inference spend. Of 23 LLM calls in the pipeline, we eliminated 9 by replacing them with deterministic code. Monthly inference cost dropped from $14,200 to $5,800. Median end-to-end latency fell by 40%. The number of production incidents tagged "non-deterministic behavior" went from 7 per month to 1. The one thing I'm still not sure about: where the line sits for tasks that are 80% deterministic and 20% genuinely ambiguous. We currently run those through the LLM, but I suspect there's a hybrid approach — deterministic fast path with LLM fallback — that we haven't built yet. If you've figured that pattern out, I'd like to hear about it. 上个季度我审计了平台的推理开支。管道中 23 个 LLM 调用里，我们通过替换为确定性代码消除了 9 个。月推理成本从 14,200 美元降到 5,800 美元。中位端到端延迟下降了 40%。标记为"不确定性行为"的生产事故从每月 7 起降到 1 起。我仍然不确定的一件事是：对于 80% 确定性、20% 真正模糊的任务，界限在哪里。我们目前把这些交给 LLM 处理，但我怀疑有一种混合方法——确定性快速路径加 LLM 回退——我们还没构建出来。如果你已经摸索出了这个模式，我很想听听。
