Your Agent Remembers Your Passwords: Building a Governance Layer for AI Memory 你的 Agent 记住了你的密码：为 AI 记忆构建治理层

AI coding assistants now have persistent memory. Claude Code writes to ~/.claude/projects/. Cursor stores context in .cursor/. Copilot retains session knowledge. This is powerful — and nobody is governing it. There is no access control, no audit trail, no retention policy, and no PII protection. This article describes why that matters, and an open-source MCP server we built to fix it. AI 编码助手现在拥有持久化记忆。Claude Code 写入 ~/.claude/projects/。Cursor 在 .cursor/ 中存储上下文。Copilot 保留会话知识。这很强大——但没有人在治理它。没有访问控制、没有审计追踪、没有留存策略、没有 PII 保护。本文描述为什么这很重要，以及我们构建的一个开源 MCP Server 来解决它。

The Problem Is Concrete, Not Hypothetical问题是具体的，不是假设的

After eight months of running AI agents in a regulated financial services environment, we cataloged four governance failures that kept recurring: 在受监管的金融服务环境中运行 AI Agent 八个月后，我们编录了四个反复出现的治理失败：

No access control. Every tool that can read the memory directory sees everything: database connection strings captured in episode logs, user behavioral corrections, compliance review notes. A debugging tool and a code generation tool have identical read access — there's no concept of "need to know." 没有访问控制。任何能读取记忆目录的工具都能看到一切：episode 日志中捕获的数据库连接字符串、用户行为纠正记录、合规审查备注。调试工具和代码生成工具有完全相同的读取权限——不存在"需要知道"的概念。

No audit trail. When a regulator asks "what did the agent know when it generated that code?", you cannot answer. The memory files exist, but there's no record of which role accessed what, when, or whether access was even authorized. In financial services, this isn't a nice-to-have — it's a compliance gap. 没有审计追踪。当监管机构问"Agent 生成那段代码时知道什么？"时，你无法回答。记忆文件存在，但没有记录哪个角色在什么时候访问了什么，甚至访问是否被授权。在金融服务中，这不是锦上添花——这是合规缺口。

No retention policy. An email address captured in a task description six months ago is still sitting in a memory file. A project context from a completed engagement is still loaded into every new session. Nothing expires, nothing archives, nothing cleans itself up. 没有留存策略。六个月前在任务描述中捕获的邮箱地址仍然存在于记忆文件中。已完成项目的上下文仍然在每个新会话中被加载。没有东西会过期、归档或自我清理。

No PII protection. When a developer writes "send the report to alice@company.com and cc bob@partner.org", both addresses are stored verbatim in the memory file. The agent didn't choose to store PII — it's an artifact of how memory capture works. But the result is the same: unredacted personal data in flat files with no access control. 没有 PII 保护。当开发者写"把报告发给 alice@company.com 并抄送 bob@partner.org"时，两个地址都原样存储在记忆文件中。Agent 并非主动选择存储 PII——这是记忆捕获工作方式的副产品。但结果是一样的：未脱敏的个人数据存在于没有访问控制的文件中。

Why Prompt-Level Governance Doesn't Work为什么提示词级别的治理不管用

The instinct is to solve this in the prompt: "Don't store PII." "Only write to project files." "Log every access." We tried this. Anthropic's 2026 agentic coding report draws a sharp line: rules that exist as text in the context achieve 70-90% compliance under normal conditions, degrading further under context pressure. Our own measurement across 47 sessions confirmed ~82% observed compliance for advisory-only governance. The 18% gap clustered around context compaction events — precisely when governance matters most. (We documented the full enforcement gap analysis, including the three-layer enforcement model, in a separate article.) 直觉的做法是在提示词中解决："不要存储 PII。""只写项目文件。""记录每次访问。"我们试过。Anthropic 2026 agentic coding 报告画了一条清晰的线：作为上下文文本存在的规则在正常条件下达到 70-90% 合规率，在上下文压力下进一步退化。我们在 47 个会话中的实测确认仅靠建议性治理的合规率约为 82%。18% 的缺口聚集在上下文压缩事件周围——恰恰是治理最重要的时候。（完整的执行缺口分析和三层执行模型见另一篇文章。）

The conclusion is structural: governance belongs at the boundary — the layer between the agent and storage — not inside the agent's reasoning. A PII filter in the prompt is advisory. A PII filter in the server is deterministic. In regulated environments, the difference between 82% and 100% is the difference between compliant and not. 结论是结构性的：治理属于边界——Agent 和存储之间的层——而不是 Agent 的推理内部。提示词中的 PII 过滤是建议性的。Server 中的 PII 过滤是确定性的。在受监管环境中，82% 和 100% 的差距就是合规与不合规的差距。

Architecture: Policy at the Boundary架构：边界处的策略

We built memory-governance, an open-source MCP server that sits between the AI agent and its memory storage. It implements the deterministic enforcement layer described in our enforcement architecture article — specifically for memory operations. MCP (Model Context Protocol) is Anthropic's standard for connecting AI tools to external data sources and operations. The server acts as a policy boundary: the agent calls tools to interact with memory, and the server enforces rules that the agent doesn't need to know about. 我们构建了 memory-governance，一个开源 MCP Server，位于 AI Agent 和其记忆存储之间。它实现了我们在执行架构文章中描述的确定性执行层——专门用于记忆操作。MCP（Model Context Protocol）是 Anthropic 的 AI 工具连接外部数据源和操作的标准。Server 充当策略边界：Agent 调用工具与记忆交互，Server 执行 Agent 不需要知道的规则。

The server exposes six tools — list_memories, read_memory, write_episode, create_memory, check_staleness, and query_audit_log — that replace direct file access with governed operations. Every call passes through four enforcement layers: Server 暴露六个工具——list_memories、read_memory、write_episode、create_memory、check_staleness 和 query_audit_log——用治理操作替代直接文件访问。每次调用都经过四个执行层：

  • RBAC access control. Roles define per-category read/write permissions. An analyst can read project memories but not behavioral feedback. A compliance_officer can read everything but write nothing. The default role can read all categories but only write to projects.RBAC 访问控制。角色定义按分类的读写权限。analyst 可以读项目记忆但不能读行为反馈。compliance_officer 可以读所有但不能写。default 角色可以读所有分类但只能写项目。
  • PII redaction. Emails, phone numbers, bearer tokens, and API keys are detected via regex and replaced with [REDACTED:type] before the data touches storage. The original values never reach disk.PII 脱敏。邮箱、手机号、Bearer Token 和 API Key 通过正则检测，在数据写入存储之前替换为 [REDACTED:type]。原始值永远不会到达磁盘。
  • Retention enforcement. Per-category max_age_days and episode_cap. When a memory file exceeds its episode cap, the oldest entries are automatically archived to a separate directory with a summary replacing them in the active file.留存策略执行。按分类设置 max_age_days 和 episode_cap。当记忆文件超过 episode 上限时，最早的条目自动归档到独立目录，在活跃文件中用摘要替代。
  • Audit logging. Every operation — reads, writes, creates, and denied accesses — is logged to an append-only JSONL file with timestamp, role, action, target, outcome, and any PII detections. The audit log has its own configurable retention (default: 2,555 days, roughly seven years, matching banking audit requirements).审计日志。每个操作——读取、写入、创建和拒绝访问——都记录到 append-only JSONL 文件中，包含时间戳、角色、操作、目标、结果和 PII 检测信息。审计日志有独立的可配置留存期（默认 2,555 天，约七年，匹配银行审计要求）。

Policy as Code策略即代码

All governance rules live in a single YAML file. Not buried in UI settings, not scattered across agent configurations — a single file that goes into version control, gets reviewed in PRs, and can be audited by compliance teams. 所有治理规则集中在一个 YAML 文件中。不是藏在 UI 设置里，不是分散在 Agent 配置中——一个文件放入版本控制，在 PR 中评审，可以被合规团队审计。 

retention:
  project:
    max_age_days: 365
    episode_cap: 10
  feedback:
    max_age_days: null      # never expires

access:
  roles:
    default:
      read: [project, area, feedback, resource]
      write: [project]
    compliance_officer:
      read: [project, area, feedback, resource]
      write: []              # read-only

pii:
  enabled: true
  redact_emails: true
  redact_phones: true
  redact_tokens: true

audit:
  retention_days: 2555       # ~7 years

This is the entire policy surface. Changing who can write to feedback memories, or how long project contexts are retained, is a one-line YAML change with a clear audit trail in git. 这就是全部的策略表面。改变谁可以写反馈记忆，或项目上下文保留多长时间，是一行 YAML 的修改，在 git 中有清晰的审计追踪。

The Memory Schema记忆 Schema

Memory files use a minimal four-field YAML frontmatter. The four-category taxonomy — project, area, feedback, resource — comes from our memory architecture, which adapts the PARA method (Projects, Areas, Resources, Archives) for AI agent memory. Each category maps to a cognitive science memory type and carries a different governance profile: 记忆文件使用最小化的四字段 YAML frontmatter。四分类体系——project、area、feedback、resource——来自我们的记忆架构，该架构将 PARA 方法（Projects、Areas、Resources、Archives）适配为 AI Agent 记忆。每个分类映射到一种认知科学记忆类型，并携带不同的治理策略： 

---
name: ap-my-project
description: One-line summary for relevance matching
category: project
updated: 2026-05-28
---

Four categories, each with a different governance profile: project (read-write, architecture decisions, episode logs), area (read-only, user profile, ongoing responsibilities), feedback (read-only, behavioral corrections), and resource (read-only, external references). The key insight: write permissions should match data volatility, not writer convenience. Project logs change constantly — make them writable. Behavioral corrections are curated — make them read-only to the agent. 四个分类，各有不同的治理策略：project（读写，架构决策，episode 日志）、area（只读，用户画像，持续职责）、feedback（只读，行为纠正）、resource（只读，外部参考）。关键洞察：写权限应匹配数据易变性，而非写入者便利性。项目日志频繁变化——设为可写。行为纠正是策展的——对 Agent 设为只读。

Zero-Install Usage via MCP通过 MCP 零安装使用

Because the server implements the MCP protocol, any compatible AI client — Claude Code, Cursor, Windsurf, or any future tool that speaks MCP — can use it without code changes. Installation is a single uvx command. Configuration is adding a JSON block to your MCP settings: 因为 Server 实现了 MCP 协议，任何兼容的 AI 客户端——Claude Code、Cursor、Windsurf，或任何未来支持 MCP 的工具——都可以无需代码修改地使用它。安装是一条 uvx 命令。配置是在 MCP 设置中添加一个 JSON 块： 

{
  "mcpServers": {
    "memory-governance": {
      "command": "uvx",
      "args": ["--from",
        "git+https://github.com/mzyag/memory-governance.git",
        "memory-governance"],
      "env": {
        "MG_POLICY_FILE": "/path/to/policy.yaml",
        "MG_MEMORY_DIR": "/path/to/memory",
        "MG_ROLE": "default"
      }
    }
  }
}

The agent now has six governed tools instead of direct file access. It doesn't know or care that PII is being redacted, that access is being logged, or that old episodes are being archived. It just calls the tools. Governance happens at the boundary, invisible to the agent but deterministic in enforcement. Agent 现在拥有六个受治理的工具而非直接文件访问。它不知道也不关心 PII 正在被脱敏、访问正在被记录、旧 episode 正在被归档。它只是调用工具。治理在边界处发生，对 Agent 不可见但在执行上是确定性的。

What It Looks Like in Practice实际效果

When an agent writes an episode containing PII, the server intercepts it before storage. The agent sends: 当 Agent 写入包含 PII 的 episode 时，Server 在存储前拦截。Agent 发送： 

write_episode(
  file: "ap-auth-refactor.md",
  goal: "Deploy auth fix, notify alice@company.com",
  outcome: "success"
)

What reaches disk: 写入磁盘的内容： 

- **2026-05-28** [success] Deploy auth fix, notify [REDACTED:email]

Meanwhile, the audit log captures the full context — without the original PII, but with the detection record: 同时，审计日志捕获完整上下文——不含原始 PII，但带有检测记录： 

{"timestamp": "2026-05-28T14:32:01Z", "action": "write",
 "role": "default", "category": "project",
 "target": "ap-auth-refactor.md", "outcome": "allowed",
 "pii_detections": ["email:alice@company.com"]}

If a compliance_officer role tries to write to the same file, the server returns a denial and logs it. No configuration change needed in the agent — the policy file controls everything: 如果 compliance_officer 角色尝试写入同一文件，Server 返回拒绝并记录。Agent 无需任何配置变更——策略文件控制一切： 

{"timestamp": "2026-05-28T14:33:15Z", "action": "write",
 "role": "compliance_officer", "category": "project",
 "target": "ap-auth-refactor.md", "outcome": "denied"}

When episode count exceeds the cap (default: 10), the oldest entries are automatically archived to a separate file, with a summary line replacing them in the active memory. This is the write-path hardening pattern described in our knowledge layer maintenance article — caps, dedup, and automatic archival to keep signal-to-noise ratio high. 当 episode 数量超过上限（默认：10）时，最早的条目自动归档到独立文件，在活跃记忆中用摘要行替代。这是我们在知识层维护文章中描述的写路径加固模式——上限、去重和自动归档以保持高信噪比。

What This Doesn't Solve (Yet)目前未解决的问题

This is v0.1 — a foundation, not a complete solution. Current limitations we're aware of: 这是 v0.1——一个基础，不是完整方案。我们已知的当前限制：

  • Regex-based PII detection. It catches standard patterns (emails, phones, tokens) but will miss novel PII formats or PII embedded in natural language. A production deployment in a high-sensitivity environment would benefit from NER-based detection.基于正则的 PII 检测。它能捕获标准模式（邮箱、手机号、token），但会遗漏新型 PII 格式或嵌入在自然语言中的 PII。在高敏感环境中的生产部署将受益于基于 NER 的检测。
  • Flat-file storage only. The governance layer is storage-agnostic by design — the policy engine doesn't care whether the backend is Markdown, SQLite, or S3. But the current implementation only ships with a flat-file store.仅支持文件存储。治理层在设计上与存储无关——策略引擎不关心后端是 Markdown、SQLite 还是 S3。但当前实现只附带文件存储。
  • No encryption at rest. Memory files are plain Markdown. For environments requiring encryption, the storage backend would need to be extended.没有静态加密。记忆文件是纯 Markdown。对于需要加密的环境，存储后端需要扩展。
  • Single-node only. No multi-agent coordination, no distributed locking, no cross-machine sync. These are real needs for team environments, and they're on the roadmap.仅单节点。没有多 Agent 协调、没有分布式锁、没有跨机器同步。这些是团队环境的真实需求，在路线图中。

Where This Fits in the Stack在技术栈中的位置

This tool is one piece of a broader agent governance stack we've been building and documenting over eight months. The full picture has three layers, each addressed by a different component: 这个工具是我们八个月来构建和记录的更广泛 Agent 治理栈中的一个组件。完整图景有三层，每层由不同组件解决：

  1. Memory architecture — how agent memory is structured, categorized, and injected into context. Covered in Memory Architecture for Production Agents.记忆架构——Agent 记忆如何结构化、分类和注入上下文。见生产级 Agent 的记忆架构。
  2. Knowledge maintenance — how memory degrades over time and how to harden the write path with caps, dedup, and TTL decay. Covered in Maintaining an Agent's Knowledge Layer.知识维护——记忆如何随时间退化，以及如何用上限、去重和 TTL 衰减加固写路径。见维护 Agent 的知识层。
  3. Governance enforcement — how to make governance deterministic rather than advisory, and why this must happen outside the context window. Covered in The Enforcement Gap. Memory-governance is the open-source implementation of this layer for memory operations.治理执行——如何让治理从建议性变为确定性，以及为什么这必须在上下文窗口之外发生。见执行缺口。Memory-governance 是这一层在记忆操作上的开源实现。

The project is open source under the MIT license: github.com/mzyag/memory-governance. Policy-as-code in YAML, local-first with no cloud dependency, and storage-agnostic by design — swap the flat-file backend for SQLite or S3 without changing a single policy rule. 项目以 MIT 许可证开源：github.com/mzyag/memory-governance。策略即代码（YAML）、本地优先无云端依赖、存储无关——把文件后端换成 SQLite 或 S3 无需改变任何策略规则。

Memory is the next attack surface in AI-assisted development. Not because agents are malicious, but because unstructured storage with unrestricted access is a governance gap waiting to be discovered — by an auditor, by a breach, or by a regulator. The right time to add governance is before any of those happen. 记忆是 AI 辅助开发中的下一个攻击面。不是因为 Agent 是恶意的，而是因为不受限制访问的非结构化存储是一个等待被发现的治理缺口——被审计师、被数据泄露、或被监管机构发现。添加治理的正确时机是在这些事情发生之前。
