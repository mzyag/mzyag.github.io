When Agents Collaborate: Failure Modes in Multi-Role LLM Systems 当 Agent 协作时：多角色 LLM 系统的失败模式

If you're splitting a task across multiple LLM agents — a planner, an executor, a reviewer — you've probably already hit the first failure mode. Maybe the executor quietly contradicted a decision the planner made for good reasons. Maybe the reviewer rejected output three times in a row and you realized the loop would never converge. Maybe you built a maker-checker pattern only to discover both roles agreed on everything, including the wrong answer. This article is about those failures: the ones that only show up when agents collaborate, not the ones you find in single-agent benchmarks. 如果你正在把一个任务拆分给多个 LLM agent——一个负责规划、一个负责执行、一个负责审查——你可能已经踩过第一个坑了。也许执行者悄悄推翻了规划者深思熟虑后做出的决策。也许审查者连续三次驳回输出，而你意识到这个循环永远不会收敛。也许你精心搭建了一个提议-复核模式，结果发现两个角色对所有东西都达成了一致——包括错误的答案。这篇文章讲的就是这类失败：只在 agent 协作时才出现的失败，而不是你在单 agent 基准测试中能发现的那些。

Why Multi-Role at All?为什么要用多角色？

A single LLM agent hits a ceiling when tasks get complex enough. Ask one agent to analyze a codebase, write a migration plan, execute the migration, and verify the result, and you'll watch it lose coherence about halfway through. It starts cutting corners on the plan because it's already thinking about execution. It skips verification because the context window is saturated with code. 当任务足够复杂时，单个 LLM agent 会遇到天花板。让一个 agent 分析代码库、编写迁移方案、执行迁移、再验证结果——你会看到它大约到一半就开始失去连贯性。它在规划阶段就开始偷工减料，因为脑子里已经在想执行的事了。它跳过验证，因为上下文窗口已经被代码塞满了。

Multi-role architectures address this by separation of concerns (the software engineering principle of isolating each part of a system to handle one responsibility). You give each agent a focused role: one plans, one executes, one reviews. Each role gets a clean context window, a clear mandate, and a bounded scope. In theory, this should produce better results than a single generalist agent — the same way a code review by a separate person catches things the author missed. 多角色架构通过关注点分离（将系统的每个部分隔离出来各自处理一项职责的软件工程原则）来解决这个问题。你给每个 agent 一个专注的角色：一个规划，一个执行，一个审查。每个角色都有干净的上下文窗口、明确的职责和有限的边界。理论上，这应该比单个通才 agent 产出更好的结果——就像由另一个人做的代码审查能发现作者遗漏的问题一样。

In practice, it does — sometimes. But it also introduces a class of coordination problems that single-agent systems never face. The failures aren't in the individual agents. They're in the spaces between them. 实践中确实如此——但只是有时候。它同时引入了一类单 agent 系统从未面对过的协调问题。故障不在单个 agent 身上，而在它们之间的缝隙里。

Failure Mode 1: Context Loss Between Roles失败模式一：角色交接时的上下文丢失

The Planner knows why a decision was made. The Executor only gets what to do. This is the most common failure mode and the most insidious, because the system appears to be working — the Executor completes the task — but the result silently violates a constraint that only the Planner was aware of. 规划者知道一个决策为什么这样做，而执行者只拿到了做什么。这是最常见、也最隐蔽的失败模式——因为系统看上去在正常工作，执行者也完成了任务——但结果悄悄违反了一个只有规划者才知道的约束条件。

Concrete example: We had a Planner that chose approach A (an in-memory cache) over approach B (a Redis-backed cache) for a data pipeline, specifically because the deployment target had a 512MB memory ceiling and the dataset was small enough to fit. The Planner's output to the Executor was: "Implement the pipeline with an in-memory cache." The reasoning — the memory constraint, the dataset size analysis, the rejection of Redis — was not passed along. Two weeks later, the dataset grew. The Executor, working on an update, hit out-of-memory errors. Without the original reasoning, it "fixed" the problem by switching to Redis — exactly the approach the Planner had evaluated and rejected for unrelated latency reasons that still applied. 具体案例：我们有一个规划者为数据管道选择了方案 A（内存缓存）而非方案 B（Redis 缓存），原因是部署目标有 512MB 的内存上限，而当时数据集小到可以放下。规划者给执行者的指令是："用内存缓存实现这个管道。"推理过程——内存约束、数据集大小分析、排除 Redis 的理由——统统没有传递。两周后数据集增长了。执行者在做更新时遇到内存溢出，在没有原始推理的情况下，它把方案"修复"成了 Redis——恰恰是规划者出于仍然成立的延迟原因而评估并否决的方案。

The handoff between roles is where institutional knowledge goes to die. Every time context is compressed into "just the instructions," you're betting that no future decision depends on information you threw away. In a multi-step task, that bet loses frequently. 角色间的交接就是机构知识消亡的地方。每次把上下文压缩成"只留指令"，你就是在赌未来不会有任何决策依赖你丢弃的信息。在多步骤任务中，这个赌注经常输。

Failure Mode 2: Role Boundary Confusion失败模式二：角色边界模糊

When does the Executor stop executing and start planning? When does the Critic stop reviewing and start rewriting? Without hard boundaries enforced by the harness (the orchestration layer that manages agent interactions), agents drift into each other's roles. LLMs are generalists — they'll happily do whatever seems helpful, regardless of their assigned role. 执行者什么时候不再执行、开始自己规划？审查者什么时候不再审查、开始自己改写？如果没有由 harness（管理 agent 交互的编排层）强制执行的硬边界，agent 会不知不觉地越界到其他角色的职责范围。LLM 是通才——只要看起来有帮助，它们不管被分配了什么角色都会照做。

Concrete example: We gave the Executor a clear mandate: "Execute the steps in the plan. Do not modify the plan." Within the first week, we found the Executor routinely adding steps it deemed "obviously necessary" — a classic "while I'm here, I should also..." pattern. In one case, the plan said "add input validation to the API endpoint." The Executor added validation, then also refactored the error handling, renamed variables for consistency, and added logging — none of which were in the plan. The Critic then had to review a diff four times larger than expected, couldn't tell which changes were planned vs. improvised, and approved the whole thing because each individual change looked reasonable. 具体案例：我们给执行者的指令很明确："按计划中的步骤执行，不要修改计划。"上线第一周我们就发现，执行者经常添加它认为"显然有必要"的步骤——典型的"既然都到这里了，顺便也......"模式。有一次，计划是"给 API 端点加输入验证"。执行者加了验证之后，又重构了错误处理、重命名了变量以保持一致性、还加了日志——这些全不在计划中。审查者随后要审查一个比预期大四倍的 diff，无法分辨哪些改动是计划内的、哪些是即兴发挥的，最终因为每项改动单独看都合理而全部通过了。

Scope creep in human teams is a management problem. Scope creep in agent teams is a systems problem — because the agent doesn't know it's creeping. It genuinely believes it's being helpful. The fix isn't better prompting; it's constraining what the agent can do at the tool level (e.g., the Executor can call file-write tools, but cannot call the planning tool). 在人类团队中，范围蔓延是管理问题。在 agent 团队中，它是系统问题——因为 agent 不知道自己在越界，它真心认为自己在帮忙。解决方法不是写更好的提示词，而是在工具层面限制 agent 能做什么（例如，执行者可以调用文件写入工具，但不能调用规划工具）。

Failure Mode 3: Infinite Rework Loops失败模式三：无限返工循环

Critic rejects. Planner revises. Executor rebuilds. Critic rejects again. Without a loop counter or convergence check (a mechanism to detect whether successive iterations are actually making progress), this cycles indefinitely. The system burns tokens, time, and — if the Executor has write access — leaves behind a trail of half-finished changes. 审查者驳回。规划者修改。执行者重建。审查者再次驳回。如果没有循环计数器或收敛检查（检测连续迭代是否真正取得进展的机制），这会无限循环下去。系统消耗 token、消耗时间——如果执行者有写权限——还会留下一堆半成品。

Concrete example: The Critic was configured with a set of quality criteria: code must have error handling, tests must pass, no hardcoded values. The Executor produced code that met all criteria — but the Critic also had an implicit standard for "code elegance" that wasn't in the criteria. It kept rejecting with feedback like "this could be more idiomatic" and "consider a more elegant approach." The Executor would refactor, producing different-but-equivalent code. The Critic would find new aesthetic objections. We discovered the loop after 7 iterations and roughly $12 in API costs for what should have been a 30-second task. 具体案例：审查者被配置了一组质量标准：代码必须有错误处理、测试必须通过、不能有硬编码值。执行者产出的代码满足了所有标准——但审查者还有一个不在标准列表中的隐性要求："代码优雅性"。它不断以"这里可以更地道""考虑一种更优雅的方式"为由驳回。执行者就重构，产出不同但等价的代码。审查者再发现新的审美问题。我们在第 7 轮迭代后才发现这个循环，一个本该 30 秒完成的任务耗费了约 12 美元的 API 费用。

The root cause is an irreconcilable gap between the Critic's standards and the Executor's capabilities. The Critic can always find something to criticize — it's an LLM, creativity is unlimited. The Executor can always produce a new version — it's an LLM, patience is unlimited. Neither agent has an incentive to stop. The loop is stable from each agent's local perspective, even though it's wasteful from a system perspective. 根本原因是审查者的标准和执行者的能力之间存在不可调和的差距。审查者总能找到可以批评的地方——它是 LLM，创造力无限。执行者总能产出新版本——它是 LLM，耐心无限。两个 agent 都没有停下来的动机。从每个 agent 的局部视角看这个循环是稳定的，尽管从系统角度看它是浪费。

Failure Mode 4: Maker-Checker Theater失败模式四：提议-复核走过场

In high-stakes decisions, you want a maker-checker pattern (also called "four-eyes principle"): one agent proposes, another independently validates. This is a well-established control in finance, aviation, and healthcare. The idea is that the checker catches errors the maker missed because the checker reasons independently. 在高风险决策中，你需要一个提议-复核模式（也叫"四眼原则"）：一个 agent 提出方案，另一个独立验证。这是金融、航空、医疗等领域行之有效的控制手段。核心思想是：复核者因为独立推理，能发现提议者遗漏的错误。

The failure: both roles use the same model, the same context, and often the same prompt structure. The checker doesn't reason independently. It reasons identically — and confirms identically. In our testing, when the maker produced a subtly wrong answer (e.g., off-by-one in a date calculation), the checker confirmed it correct 89% of the time. The checker wasn't checking; it was echoing. "Independent verification" with shared reasoning isn't verification. It's theater. 失败之处在于：两个角色用同一个模型、同一份上下文，往往还用相同的提示词结构。复核者不是独立推理，而是以相同方式推理——然后以相同方式确认。在我们的测试中，当提议者产出一个微妙的错误答案（比如日期计算差一天）时，复核者有 89% 的概率确认它是正确的。复核者没有在复核；它在回声。用共享推理做"独立验证"不是验证，是走过场。

This is particularly dangerous because it creates a false sense of security. The team sees "two agents agreed" and assumes the answer was validated. In reality, one agent was wrong and the other was a mirror. We call this "consensus without independence" — the multi-agent equivalent of copying someone's homework and both submitting the same wrong answer. 这特别危险，因为它制造了虚假的安全感。团队看到"两个 agent 达成一致"就以为答案经过了验证。实际上是一个 agent 搞错了，另一个只是它的镜像。我们称之为"无独立性的共识"——多 agent 版本的抄作业，两个人交了同一份错误答案。

Failure Mode 5: Arbitration Deadlocks失败模式五：仲裁死锁

Two agents disagree. Who wins? This question sounds simple until you realize there's no good default answer. Without explicit arbitration rules (coded decision criteria that determine how conflicts are resolved), the system either picks randomly, defers to the last speaker, or escalates everything to a human — which defeats the purpose of automation. 两个 agent 意见不一致。听谁的？这个问题听起来简单，直到你意识到没有一个好的默认答案。如果没有明确的仲裁规则（确定如何解决冲突的编码决策标准），系统要么随机选一个、要么听最后发言者的、要么所有东西都升级给人类——这就失去了自动化的意义。

Concrete example: The Planner said "use approach A — a batch processing strategy." The Critic reviewed the Executor's implementation and said "approach A has scalability risk X under peak load; use approach B — a streaming strategy." Both were right within their own reasoning. The Planner's analysis was based on average load patterns. The Critic's analysis was based on worst-case scenarios. Without coded decision criteria — such as "if the dispute involves load handling, prioritize worst-case analysis" — the system entered an LLM debate. The Planner defended its choice. The Critic defended its objection. They went back and forth for 4 rounds, generating progressively longer arguments, before the human operator noticed and intervened. 具体案例：规划者说"使用方案 A——批处理策略"。审查者审查了执行者的实现后说"方案 A 在峰值负载下有可扩展性风险 X，应使用方案 B——流处理策略"。在各自的推理框架内两者都是对的：规划者的分析基于平均负载模式，审查者的分析基于最坏情况。没有编码的决策标准（比如"如果争议涉及负载处理，优先考虑最坏情况分析"），系统进入了 LLM 辩论。规划者为自己的选择辩护，审查者为自己的反对辩护。它们来回辩了 4 轮，论述越来越长，直到人类操作员注意到并介入。

LLMs are very good at generating arguments. They are very bad at converging on a decision. Without external arbitration logic, agent debates are stable — both sides will keep producing reasonable-sounding arguments indefinitely. This is different from the rework loop: in a rework loop, the Executor is trying to satisfy the Critic. In an arbitration deadlock, neither agent is trying to satisfy the other. They're both trying to persuade. LLM 非常擅长生成论据，却非常不擅长收敛到一个决策。没有外部仲裁逻辑，agent 辩论是稳态的——双方会无限期地继续产出听起来合理的论点。这和返工循环不同：在返工循环中，执行者在努力满足审查者的要求；在仲裁死锁中，两个 agent 都不是在试图满足对方，而是都在试图说服对方。

What We Did About It我们怎么解决的

None of these have perfect solutions. But each has mitigations that reduced the failure rate from "constant" to "occasional." Here's what worked for us: 这些问题没有一个有完美解决方案。但每一个都有缓解措施，把故障率从"频繁"降到了"偶发"。以下是对我们有效的做法：

Context Loss: Structured Handoff Schema上下文丢失：结构化交接协议

We stopped passing freeform text between roles. Instead, every handoff uses a structured schema that carries not just the plan, but the reasoning behind it. The Planner's output includes: the chosen approach, the alternatives considered, the reasons each alternative was rejected, and the constraints that influenced the decision. The Executor receives all of this as immutable context. It can't modify the reasoning, but it can reference it when making execution-level decisions. This didn't eliminate context loss entirely — you can't force the Executor to read the reasoning — but it made the information available when it mattered. 我们不再在角色之间传递自由格式的文本。每次交接都使用一个结构化 schema，不仅包含计划本身，还包含背后的推理。规划者的输出包括：选定的方案、考虑过的备选方案、每个备选方案被否决的原因、以及影响决策的约束条件。执行者以不可变上下文的形式接收这一切——它不能修改推理，但可以在做执行层面的决策时参考。这没有完全消除上下文丢失——你无法强制执行者阅读推理——但在关键时刻信息是可用的。

Role Confusion: Hard Boundaries at the Tool Level角色越界：工具层面的硬边界

Prompts saying "stay in your role" don't work reliably. What does work: constraining the agent's available tools. The Executor can call file-write, shell-execute, and test-run tools. It cannot call the planning tool, the architecture tool, or the refactoring tool. If the Executor decides it needs to modify the plan, it has no mechanism to do so — it can only flag the issue and request a rework cycle. The Critic can produce structured review feedback (approve, reject with reason, request clarification). It cannot call any write tools. If it wants to "fix" the code, it has to express the fix as feedback that the Executor implements. This is the difference between a policy and a constraint. Policies rely on the agent's compliance. Constraints are enforced by the system. 提示词说"保持你的角色"并不可靠。有效的做法是：限制 agent 可用的工具。执行者可以调用文件写入、shell 执行和测试运行工具，不能调用规划工具、架构工具或重构工具。如果执行者认为需要修改计划，它没有机制去做——只能标记问题、请求返工周期。审查者可以产出结构化审查反馈（通过、附理由驳回、请求澄清），不能调用任何写入工具。如果它想"修复"代码，必须把修复表述为反馈，由执行者来实施。这就是策略和约束的区别：策略依赖 agent 的遵守；约束由系统强制执行。

Rework Loops: Per-Issue Retry Limits返工循环：按问题设重试上限

We added a per-issue retry counter: each specific rejection reason gets at most 2 rework attempts. After 2 failed attempts on the same issue, the system escalates to a human with the full conversation history. This means the Critic can still reject — but not indefinitely. If the Executor can't satisfy a particular criterion in 2 tries, the gap is probably structural, not fixable by iteration. We also added a convergence check: if the Executor's output diff between iterations is below a threshold (meaning it's producing near-identical code each time), we short-circuit the loop and escalate immediately. This catches the "different but equivalent" pattern where the Executor is just rearranging the same code. 我们加了按问题的重试计数器：每个具体的驳回理由最多允许 2 次返工。同一问题 2 次失败后，系统携带完整对话历史升级给人类。这意味着审查者仍然可以驳回——但不能无限驳回。如果执行者在 2 次尝试中无法满足某项标准，差距可能是结构性的，迭代解决不了。我们还加了收敛检查：如果执行者在两次迭代之间的输出 diff 低于阈值（意味着每次产出的代码几乎相同），就立即短路循环并升级。这能捕捉到执行者只是在重新排列相同代码的"不同但等价"模式。

Maker-Checker Theater: Break the Symmetry复核走过场：打破对称性

For the checker to actually catch errors, it needs to reason differently from the maker. We found three approaches that helped, in order of effectiveness: 要让复核者真正抓到错误，它需要以不同于提议者的方式推理。我们发现了三种有帮助的方法，按效果排序：

  1. Different models. Use a different model for the checker than the maker. Different models have different failure modes. A mistake that one model makes consistently, another might catch. This was the single most effective change.不同的模型。复核者和提议者使用不同的模型。不同模型有不同的失败模式。一个模型一贯会犯的错，另一个可能会发现。这是最有效的单项改进。
  2. Different context. Don't give the checker the maker's reasoning. Give it the task, the output, and ask it to find problems. If it sees the maker's chain of thought, it follows the same path. If it starts fresh, it might take a different path.不同的上下文。不要把提议者的推理过程给复核者看。只给它任务和输出，让它找问题。如果看到提议者的思维链，它会沿着同一条路走。如果从头开始，它可能走不同的路。
  3. Evidence requirement. The checker must cite evidence not present in the maker's output. "I agree this looks correct" is not valid checker output. "I verified X by checking Y, which confirms Z" is valid. This forces the checker to do actual work, not just pattern-match against the maker's reasoning.证据要求。复核者必须引用提议者输出中不存在的证据。"看起来没问题"不是有效的复核输出，"我通过检查 Y 验证了 X，确认 Z"才是。这迫使复核者做实际工作，而不是仅仅对提议者的推理做模式匹配。

Arbitration Deadlocks: Coded Rules, Not LLM Debate仲裁死锁：编码规则而非 LLM 辩论

When two agents disagree, the resolution should not involve more LLM inference. We implemented a simple arbitration framework: 当两个 agent 意见不一致时，解决方式不应该涉及更多 LLM 推理。我们实现了一个简单的仲裁框架：

  • Typed disagreements. Every disagreement is classified into a type: correctness, performance, style, security. Each type has a coded resolution rule.类型化分歧。每个分歧被分类为一种类型：正确性、性能、风格、安全。每种类型有编码的解决规则。
  • Priority hierarchy. Security concerns override performance concerns. Correctness overrides style. This is coded, not prompted.优先级层次。安全问题优先于性能问题，正确性优先于风格。这是编码的，不是提示词中说的。
  • Human as final arbiter. If the disagreement doesn't fall into a coded category, or if both sides have equal-priority concerns, the system escalates to a human with structured context: what each agent said, what type of disagreement it is, and what the coded rules would suggest if they applied.人类是最终仲裁者。如果分歧不属于任何编码类别，或者双方的关切优先级相同，系统携带结构化上下文升级给人类：每个 agent 说了什么、属于什么类型的分歧、以及编码规则适用时会给出什么建议。

The key insight: arbitration is not an AI problem. It's a governance problem. The system needs explicit, auditable decision-making rules — the same way an organization needs a decision-rights matrix, not a meeting where everyone argues until someone gives up. 关键洞察：仲裁不是 AI 问题，是治理问题。系统需要明确、可审计的决策规则——就像一个组织需要决策权限矩阵，而不是一个所有人争论到有人放弃的会议。

Checklist: Multi-Agent Collaboration清单：多 Agent 协作

  1. Carry reasoning, not just instructions. Every handoff between roles should include the "why" alongside the "what." Use a structured schema, not freeform text. If you can't explain why a decision was made, the next agent can't respect that decision.传递推理，不只是指令。每次角色间的交接都应包含"为什么"和"做什么"。使用结构化 schema，不要用自由格式文本。如果你解释不了为什么做了某个决定，下一个 agent 就无法尊重这个决定。
  2. Enforce role boundaries with tools, not prompts. Each agent should only have access to the tools its role requires. The Executor cannot call the planning tool. The Critic cannot call write tools. Constraints are more reliable than instructions.用工具约束角色边界，而非提示词。每个 agent 只应能访问其角色所需的工具。执行者不能调用规划工具，审查者不能调用写入工具。约束比指令更可靠。
  3. Add retry limits and convergence checks. Every feedback loop needs a counter. Per-issue, not global — otherwise one easy fix uses up the retry budget for a hard problem. Add convergence detection: if two consecutive iterations produce near-identical output, the loop isn't making progress.加上重试限制和收敛检查。每个反馈循环都需要计数器。按问题计数，不是全局的——否则一个简单修复会耗尽困难问题的重试配额。加收敛检测：如果连续两次迭代产出近乎相同的输出，循环没有在进步。
  4. Break maker-checker symmetry. If two agents serve as maker and checker, they must differ in at least one dimension: different models, different context, or different evidence requirements. Identical reasoning produces identical errors.打破提议-复核的对称性。如果两个 agent 分别担任提议者和复核者，它们至少要在一个维度上不同：不同模型、不同上下文、或不同的证据要求。相同的推理产生相同的错误。
  5. Code your arbitration rules. When agents disagree, the resolution mechanism should be deterministic code, not another LLM call. Classify disagreements by type, define priority hierarchies, and escalate what can't be resolved automatically.编码你的仲裁规则。当 agent 意见不一致时，解决机制应该是确定性代码，不是另一次 LLM 调用。按类型分类分歧、定义优先级层次、把无法自动解决的升级给人类。
  6. Instrument the boundaries. Log every handoff, every rework cycle, every disagreement. The failures in multi-agent systems are at the boundaries — if you're not observing the boundaries, you're flying blind.为边界加装观测。记录每次交接、每个返工周期、每次分歧。多 agent 系统的故障在边界上——如果你不观测边界，就是在盲飞。
  7. Default to escalation, not autonomous resolution. When you're unsure whether the system can handle a situation, escalate to a human. It's better to over-escalate early than to silently produce wrong output. You can always reduce escalation rates later once you understand the patterns.默认升级，而非自主解决。当你不确定系统能否处理某种情况时，升级给人类。早期过度升级好过悄悄产出错误结果。一旦你理解了模式，之后总可以降低升级率。

What We Haven't Solved尚未解决的问题

I want to be honest about the gaps. The mitigations above reduced failures, but they didn't eliminate them. 我想坦诚地谈谈这些缺口。上面的缓解措施减少了故障，但没有消除。

Context loss is still the biggest problem. Structured handoff schemas help, but they can't capture everything the Planner considered. Some context is implicit — the Planner explored a dead end and the very fact of that exploration informed its final choice, but the dead end wasn't interesting enough to document. The Executor doesn't know what it doesn't know. 上下文丢失仍然是最大的问题。结构化交接协议有帮助，但无法捕获规划者考虑过的所有东西。有些上下文是隐含的——规划者探索了一条死路，这个探索过程本身影响了最终选择，但那条死路不值得专门记录。执行者不知道自己不知道什么。

Maker-checker still fails on novel errors. Using different models helps with known error patterns. But when both models encounter a genuinely novel edge case, they can both get it wrong in the same way — not because they share reasoning, but because the error is genuinely hard. Different models help with systematic biases, not with the frontier of difficulty. 提议-复核在新颖错误上仍然失效。使用不同模型有助于应对已知的错误模式。但当两个模型都遇到一个真正新颖的边界情况时，它们可能以相同的方式犯错——不是因为共享推理，而是因为错误本身就很难。不同模型有助于应对系统性偏差，但不能应对难度的前沿。

We don't have a good theory for optimal role decomposition. How many roles should a multi-agent system have? Where should the boundaries be? We made these decisions based on intuition and iteration, not theory. Three roles (Planner, Executor, Critic) worked well for our tasks, but we don't know if it's optimal, and adding a fourth role (e.g., a separate Verifier) introduced more coordination overhead than it eliminated errors. 我们没有关于最优角色分解的好理论。一个多 agent 系统应该有多少个角色？边界应该画在哪里？我们基于直觉和迭代做出了这些决定，而非理论。三个角色（规划者、执行者、审查者）在我们的任务上效果不错，但不知道是否最优。尝试加第四个角色（比如单独的验证者）时，引入的协调开销比它消除的错误更多。

Multi-agent systems are not fundamentally harder than single-agent systems. They're harder in different places. Single-agent systems fail on complexity — one agent can't hold everything in context. Multi-agent systems fail on coordination — the agents can each be competent, and the system as a whole still produces wrong output because something was lost between them. If you're building a multi-agent system, spend less time optimizing individual agents and more time instrumenting the spaces between them. That's where the bugs live. 多 agent 系统并不从根本上比单 agent 系统更难，只是难在不同的地方。单 agent 系统在复杂度上失败——一个 agent 无法把所有东西放在上下文中。多 agent 系统在协调上失败——每个 agent 都可以很有能力，但系统整体仍然产出错误输出，因为有些东西在它们之间丢失了。如果你在构建多 agent 系统，少花时间优化单个 agent，多花时间观测它们之间的缝隙。Bug 就住在那里。
