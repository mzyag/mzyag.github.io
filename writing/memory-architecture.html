Memory Architecture for Production Agents: Beyond the Context Window 生产级 Agent 的记忆架构：超越上下文窗口

If you've told your agent "always use /workspace as the project root" on Monday only to have it ask again by Wednesday, you've hit the goldfish problem. Most agents have no memory beyond the current context window. In a demo, that's an inconvenience. In production, it's a trust destroyer — users stop relying on the agent because they can't trust it to remember what they've already established. This article walks through a three-tier memory architecture, a four-dimension content taxonomy, budget-aware context injection, and an episode-based self-improvement loop, all drawn from eight months of production operation on a financial services platform. 如果你周一告诉 Agent "始终使用 /workspace 作为项目根目录"，到周三它又在问你，你就遇到了"七秒记忆"问题。大多数 Agent 的记忆不会超出当前上下文窗口。演示时这只是小瑕疵。到了生产，这直接摧毁用户信任——用户发现 Agent 记不住已经说好的事情，就不会再依赖它。本文将介绍一个三层记忆架构、四维度内容分类、预算感知的上下文注入以及基于 episode 的自改进循环，全部来自在金融服务平台上八个月的生产运行经验。

Key Takeaways核心要点

  • Persistent memories have higher per-token value than conversation history and should never be auto-trimmed; if context budget forces a choice, conversation turns are evicted first.持久记忆的每 token 价值高于对话历史，不应被自动裁剪；当上下文预算迫使选择时，优先驱逐对话轮次。
  • A four-dimension content taxonomy (User, Feedback, Project, Reference) maps to cognitive science memory types and gives the system structured metadata for injection priority, staleness detection, and conflict resolution without requiring LLM reasoning.四维度内容分类（User、Feedback、Project、Reference）映射到认知科学记忆类型，为系统提供结构化元数据以决定注入优先级、过期检测和冲突解决，无需 LLM 推理。
  • A one-hour session TTL with compression to a one-paragraph summary is the empirical sweet spot: long enough to survive coffee breaks, short enough to prevent yesterday's context from polluting today's work.一小时的会话 TTL 加压缩为一段摘要是经验最优点：足以覆盖一次休息，又不会让昨天的上下文污染今天的工作。
  • Feedback memories — user corrections and validated approaches — are the highest-priority injection category because missing one means repeating a mistake the user already told you about.Feedback 记忆——用户纠正和已验证方法——是最高优先级的注入类别，因为遗漏一条意味着重复用户已经告知的错误。
  • Self-improvement should be human-driven: episode logs provide structured evidence, but automated root cause analysis at 70% accuracy means 30% of fixes silently degrade the system.自改进应由人驱动：episode 日志提供结构化证据，但 70% 准确率的自动化根因分析意味着 30% 的修复在悄悄降低系统质量。

Why Sliding Windows Aren't Enough为什么滑动窗口不够

The simplest memory implementation is a sliding window: keep the last N turns of conversation, drop the oldest when you exceed the limit. This is what most agent frameworks do by default, and it works for short, self-contained interactions. 最简单的记忆实现是滑动窗口：保留最近 N 轮对话，超出限制时丢弃最旧的。这是大多数 Agent 框架的默认行为，对于短的、自包含的交互有效。

It fails in three ways: 它在三个方面失败：

  • Preferences evaporate. A user says "reply in Chinese" in turn 3. By turn 15, that instruction has fallen out of the window. The agent switches back to English with no warning. The user has to repeat themselves — and each repetition erodes trust.偏好蒸发。用户在第 3 轮说"用中文回复"。到第 15 轮，这条指令已经滑出窗口。Agent 毫无预警地切回英文。用户不得不重复——每次重复都在侵蚀信任。
  • Context is all-or-nothing. A 20-turn conversation about refactoring the auth module either occupies 20 turns of context (expensive) or is completely forgotten (useless). There's no middle ground — no way to say "remember we discussed auth refactoring, forget the details."上下文是全有或全无的。一段关于重构认证模块的 20 轮对话要么占据 20 轮上下文（昂贵），要么完全被遗忘（无用）。没有中间地带——没有办法说"记住我们讨论过认证重构，忘记细节"。
  • No cross-session learning. The agent completes a task, the session ends, and everything learned during that task is gone. Next session, same mistakes, same clarifications, same ramp-up time.没有跨会话学习。Agent 完成一个任务，会话结束，任务期间学到的一切都消失了。下一次会话，同样的错误，同样的澄清，同样的预热时间。

The fundamental problem: not all information has the same lifetime or the same importance. A preference ("reply in Chinese") should live forever. A conversation detail ("we discussed renaming the function to processOrder") should live for hours. A task outcome ("refactoring succeeded, 8 files changed") should be compressed and archived. Treating all information the same — which a sliding window does — guarantees that you'll either waste tokens on stale details or lose critical preferences. 根本问题：不是所有信息都有相同的生命周期或相同的重要性。偏好（"用中文回复"）应该永远存在。对话细节（"我们讨论了将函数重命名为 processOrder"）应该存活几小时。任务结果（"重构成功，8 个文件更改"）应该被压缩和归档。同等对待所有信息——滑动窗口就是这样做的——保证你要么在过时细节上浪费 token，要么丢失关键偏好。

Three-Tier Memory Architecture三层记忆架构

We designed a memory system with three tiers, each with a different storage policy, lifetime, and priority: 我们设计了一个三层记忆系统，每层有不同的存储策略、生命周期和优先级：

Tier 1: Conversation (Working Memory)第一层：对话（工作记忆）

Recent conversation turns, stored in-memory. This is the hot path — the most recent exchanges that the agent needs for immediate context. Default window: last 6 turns. 最近的对话轮次，存储在内存中。这是热路径——Agent 需要用于即时上下文的最近交互。默认窗口：最近 6 轮。

The critical addition: session TTL. If the user is inactive for more than one hour, the conversation window expires. But it doesn't just disappear — it gets compressed into a summary (Tier 3) before eviction. The next time the user speaks, the agent doesn't have the raw conversation, but it has a compressed digest: "Last session: we discussed refactoring task_bot.py into 8 modules. Outcome: completed successfully." 关键新增：会话 TTL。如果用户超过一小时不活跃，对话窗口过期。但它不会直接消失——在驱逐前被压缩为摘要（第三层）。下次用户说话时，Agent 没有原始对话，但有压缩摘要："上次会话：我们讨论了将 task_bot.py 拆分为 8 个模块。结果：成功完成。"

Why one hour? Shorter than that, and you lose context during natural breaks (lunch, meetings). Longer, and you carry stale context that wastes tokens and confuses the agent. One hour is the sweet spot we found empirically — long enough for "I stepped away to grab coffee" but short enough that yesterday's conversation doesn't pollute today's context. 为什么是一小时？比这短，你会在自然中断（午餐、会议）期间丢失上下文。比这长，你会携带浪费 token 并混淆 Agent 的陈旧上下文。一小时是我们反复调参后找到的平衡点——"出去倒杯咖啡"回来上下文还在，但不至于把昨天的对话带进今天。

Tier 2: Persistent Memory (Long-Term)第二层：持久记忆（长期）

Persistent facts extracted from conversations. These survive session resets, process restarts, and explicit /reset commands. When a user says "clear my chat history," they mean the conversation — not "forget what you've learned about how I work." 从对话中提取的持久事实。它们在会话重置、进程重启和显式 /reset 命令后保留。当用户说"清除我的聊天记录"时，他们指的是对话——不是"忘记你了解的我的工作方式"。

Extraction is signal-based: explicit cues ("always," "from now on," "remember this") and implicit cues (corrections, role disclosures, resource mentions) trigger persistent memory writes. But not all persistent memories are the same — we discovered early that lumping everything into a single "preferences" bucket creates retrieval, staleness, and conflict problems. The solution is a four-dimension content taxonomy, covered in the next section. 提取是基于信号的：显式线索（"始终"、"从现在开始"、"记住这个"）和隐式线索（纠正、角色透露、资源提及）触发持久记忆写入。但并非所有持久记忆都一样——我们很早就发现把一切塞进单一的"偏好"桶会导致检索、过期和冲突问题。解决方案是四维度内容分类，在下一节详述。

Persistent memories are never auto-trimmed. They're the highest-priority memory tier. If the token budget forces a choice between keeping a persistent memory and keeping a conversation turn, the conversation turn goes. The key insight: persistent memories have higher per-token value than conversation history — they're reusable knowledge, not ephemeral details. If something has to go, it's the conversation turn. 持久记忆永不自动裁剪。它们是最高优先级的记忆层。如果 token 预算迫使在保留持久记忆和保留对话轮次之间选择，对话轮次被丢弃。这是关键洞察：持久记忆的每 token 价值高于对话历史，因为它们编码可复用的知识而非短暂的细节。如果必须舍弃什么，舍弃的是对话轮次。

An obvious question: why not use embedding-based semantic search for memory retrieval, the way RAG retrieves documents? Three reasons. First, the corpus is small — typically 20–50 records per user — so a linear scan is sub-millisecond; vector indexing adds complexity without speed gain. Second, embedding similarity is unreliable for short directive memories ("always reply in Chinese") — these are instructions, not documents, and embedding models rank them poorly against diverse queries. Third, we inject all feedback and user memories unconditionally (they're compact and high-priority), so retrieval accuracy only matters for lower-priority dimensions where structured field matching (type, scope, recency) outperforms semantic similarity. If the corpus grew past ~500 records per user, we'd revisit this — but at current scale, keyword + structured metadata beats embeddings. 一个显而易见的问题：为什么不像 RAG 检索文档那样，用基于 embedding 的语义搜索来检索记忆？三个原因。第一，语料库很小——通常每用户 20-50 条——线性扫描亚毫秒完成，向量索引增加复杂度却没有速度收益。第二，embedding 相似度对短指令性记忆（"始终用中文回复"）不可靠——这些是指令而非文档，embedding 模型在面对多样化查询时排名很差。第三，我们无条件注入所有 Feedback 和 User 记忆（它们紧凑且高优先级），检索准确性只对低优先级维度有意义，而在那里结构化字段匹配（类型、范围、时效性）优于语义相似度。如果语料库增长到每用户约 500 条以上，我们会重新评估——但在当前规模，关键词 + 结构化元数据优于 embedding。

Tier 3: Summaries (Compressed Memory)第三层：摘要（压缩记忆）

When a conversation window expires (TTL hit), the raw turns are compressed into a one-paragraph summary. We keep the most recent 10 summaries. This gives the agent a "notebook" of past work — enough to say "we've worked on this before" without carrying the full transcript. 当对话窗口过期（TTL 触发）时，原始轮次被压缩为一段摘要。我们保留最近 10 条摘要。这给 Agent 一个过去工作的"笔记本"——足以说"我们以前处理过这个"而不携带完整记录。

Compression can be LLM-powered ("summarize this conversation in 2 sentences focusing on decisions made and outcomes") or, when LLM summarization is unavailable, a simple truncation fallback that keeps the first and last turn plus any turns containing keywords like "decided," "confirmed," "completed." 压缩可以由 LLM 驱动（"用 2 句话总结这段对话，聚焦于做出的决策和结果"），或者当 LLM 总结不可用时，使用简单的截断降级——保留第一轮和最后一轮，加上包含"决定"、"确认"、"完成"等关键词的轮次。

Content Taxonomy: Four Dimensions of Persistent Memory内容分类：持久记忆的四个维度

The three tiers describe when memory lives and dies. But "preferences" as a single category for all long-term memory is too coarse. In production, we found that persistent memories differ not just in lifetime but in what kind of knowledge they encode. We identified four dimensions, each with different storage patterns, update triggers, and retrieval priorities. 三层架构描述了记忆何时生存和消亡。但把所有长期记忆归为"偏好"这个单一类别太粗了。在生产中，我们发现持久记忆的差异不仅在生命周期，还在于它们编码的知识类型。我们识别出四个维度，每个都有不同的存储模式、更新触发和检索优先级。

This isn't a novel insight — it maps to well-established memory models. In cognitive science, Tulving's classification (1972) distinguishes semantic memory (facts about the world), episodic memory (events and experiences), and procedural memory (learned behaviors). In agent systems, MemGPT (Packer et al., NeurIPS 2023) formalizes a split between core memory — always loaded, bounded, critical (user profile and agent persona) — and archival memory — searched on demand, unbounded. LangChain's LangMem framework uses a similar three-way split: semantic (facts about users), episodic (conversation history), and procedural (learned instructions). Our four dimensions map to these frameworks: 这不是全新的洞察——它映射到成熟的记忆模型。在认知科学中，Tulving 的分类（1972）区分了语义记忆（关于世界的事实）、情景记忆（事件和经历）和程序记忆（习得的行为）。在 Agent 系统中，MemGPT（Packer et al., NeurIPS 2023）形式化了核心记忆（始终加载、有界、关键——用户画像和 Agent 人格）与归档记忆（按需搜索、无界）的分离。LangChain 的 LangMem 框架使用类似的三分法：语义（关于用户的事实）、情景（对话历史）和程序（习得的指令）。我们的四个维度映射到这些框架：

  • User + Reference → Semantic memory. Facts about the world — who the user is, where resources live. We split these because they have different staleness profiles: a user's role rarely changes; a Notion URL might move.User + Reference → 语义记忆。关于世界的事实——用户是谁、资源在哪里。我们将它们分开是因为过期特征不同：用户角色很少变化；Notion URL 可能迁移。
  • Feedback → Procedural memory. Learned behaviors and instructions — what to do, what not to do, and why. This is the dimension most agent systems under-invest in.Feedback → 程序记忆。习得的行为和指令——该做什么、不该做什么、以及为什么。这是大多数 Agent 系统投入不足的维度。
  • Project → Contextual/episodic memory. Current state of the world — ongoing work, deadlines, decisions. High decay rate, needs active staleness management.Project → 情境/情景记忆。世界的当前状态——正在进行的工作、截止日期、决策。衰减率高，需要主动的过期管理。
  • Tier 1 conversation + Tier 3 summaries → Episodic memory. Already handled by the three-tier architecture above.第一层对话 + 第三层摘要 → 情景记忆。已由上面的三层架构处理。

Following MemGPT's core/archival distinction, we further classify the four dimensions into two tiers: core memory (Feedback + User) — always injected, compact, high per-token value — and contextual memory (Project + Reference) — injected when budget allows, searched by recency and relevance. This isn't just a priority list; it's a design contract. Core memory has a hard size cap (we enforce 200 chars per User record, 500 chars total for Feedback) to guarantee it always fits. Contextual memory is elastic — it fills whatever budget remains. 遵循 MemGPT 的核心/归档区分，我们进一步将四个维度分为两级：核心记忆（Feedback + User）——始终注入、紧凑、每 token 价值高——和情境记忆（Project + Reference）——预算允许时注入，按时效和相关性搜索。这不只是优先级列表；它是设计契约。核心记忆有硬性大小上限（我们限定每条 User 记录 200 字符、Feedback 总计 500 字符）以保证始终放得下。情境记忆是弹性的——填充剩余预算。

The four dimensions: 四个维度：

Dimension 1: User — Who Am I Talking To?维度一：User — 我在和谁对话？

Role, expertise level, responsibilities, and domain knowledge. A senior distributed systems engineer asking "explain this retry logic" wants a different answer than an intern asking the same question. Without user memory, the agent defaults to the same generic explanation every time. 角色、专业水平、职责和领域知识。一个资深分布式系统工程师问"解释这段重试逻辑"和实习生问同样的问题，需要不同的回答。没有 User 记忆，Agent 每次都给出同样的通用解释。

When to save: When the user reveals their role ("I'm a data scientist"), expertise ("I've been writing Go for ten years but this is my first time with React"), or responsibility ("I own the payment service"). How it changes behavior: The agent adapts explanation depth, assumes domain knowledge where appropriate, and frames recommendations in terms the user already understands. A backend engineer gets architecture analogies; a product manager gets impact framing. 何时保存：当用户透露角色（"我是数据科学家"）、专业度（"我写了十年 Go 但第一次接触 React"）或职责（"我负责支付服务"）时。如何影响行为：Agent 调整解释深度，在适当时假设领域知识，用用户已理解的术语组织建议。后端工程师得到架构类比；产品经理得到影响力框架。 

{
  "type": "user",
  "content": "Senior backend engineer, deep Go expertise, new to React",
  "source": "user message at 2025-03-15T09:10:00"
}

Dimension 2: Feedback — What Should I Do Differently?维度二：Feedback — 我应该怎么调整？

Corrections the user has made and approaches they've validated. This is the most important dimension for avoiding repeated mistakes. If a user says "don't mock the database in these tests — we got burned last quarter when mocked tests passed but the prod migration failed," that's a feedback memory. But equally important: if the user confirms "yes, the single bundled PR was the right call here," that's also feedback — a validated approach worth repeating. 用户做过的纠正和验证过的方法。这是避免重复犯错最重要的维度。如果用户说"这些测试不要 mock 数据库——上季度 mock 测试通过了但生产迁移失败了"，这是 Feedback 记忆。同样重要的是：如果用户确认"没错，这次合成一个 PR 是对的"，这也是 Feedback——一个值得重复的已验证方法。

When to save: Any time the user corrects your approach ("no, not that way") OR confirms a non-obvious approach worked. Corrections are easy to notice; confirmations are quieter — you have to watch for them. Key design decision: Each feedback record stores the rule, the why behind it, and when to apply it. Without the why, the agent follows rules blindly and can't judge edge cases. 何时保存：用户纠正方法时（"不，别这样做"）或确认非显而易见的方法有效时。纠正容易注意到；确认更安静——你得主动去捕捉。关键设计决策：每条 Feedback 记录存储规则、背后的原因和何时应用。没有原因，Agent 会盲目遵循规则而无法判断边界情况。 

{
  "type": "feedback",
  "content": "Integration tests must hit a real database, not mocks",
  "why": "Prior incident: mock/prod divergence masked a broken migration",
  "apply_when": "Writing or reviewing test code for database-dependent features",
  "source": "user correction at 2025-03-18T14:30:00"
}

Dimension 3: Project — What's Happening Right Now?维度三：Project — 现在在发生什么？

Ongoing work, goals, decisions, deadlines, and context that isn't derivable from the codebase alone. A git log tells you what changed. Project memory tells you why it changed and what's coming next. "We're freezing all non-critical merges after Thursday — mobile team is cutting a release branch" is a project memory. So is "the auth middleware rewrite is driven by legal compliance requirements, not tech debt." 正在进行的工作、目标、决策、截止日期和仅从代码库无法推导的上下文。Git log 告诉你什么改了。Project 记忆告诉你为什么改了和接下来要做什么。"周四之后冻结所有非关键合并——移动端团队要切发布分支"是 Project 记忆。"认证中间件重写是因为法规合规要求，不是技术债"也是。

When to save: When you learn who is doing what, why, or by when. Critical rule: Always convert relative dates to absolute dates when saving. "Next Thursday" becomes "2025-03-20." Otherwise the memory becomes uninterpretable after a week. Staleness management: Project memories go stale faster than any other type. A merge freeze from three weeks ago is probably over. We handle this with two mechanisms borrowed from knowledge management practice: a last_verified timestamp that records when the memory was last confirmed still true, and a confidence decay function — memories older than 14 days without re-verification are injected with a staleness warning ("⚠ unverified since {date}"), and memories older than 30 days are demoted below Reference priority. The agent should treat unverified Project memories as context for what was true, not what is true, and check the current state (git log, issue tracker, team chat) before acting on them. 何时保存：了解到谁在做什么、为什么、什么时候截止时。关键规则：保存时始终将相对日期转换为绝对日期。"下周四"变成"2025-03-20"。否则一周后记忆就无法解读了。过期管理：Project 记忆比其他任何类型都更快过期。三周前的合并冻结可能已经结束了。我们借鉴知识管理实践，用两个机制处理：一个 last_verified 时间戳记录记忆上次被确认仍然为真的时间，以及一个置信衰减函数——超过 14 天未被重新验证的记忆在注入时附带过期警告（"⚠ 未验证，自 {date} 起"），超过 30 天的记忆被降级到 Reference 优先级以下。Agent 应将未验证的 Project 记忆视为曾经为真的上下文，而非现在为真，并在据此行动前检查当前状态（git log、issue tracker、团队聊天）。

Dimension 4: Reference — Where Do I Look?维度四：Reference — 去哪里找？

Pointers to where information lives in external systems. Not the information itself — just the address. "Pipeline bugs are tracked in Linear project INGEST." "The oncall latency dashboard is at grafana.internal/d/api-latency." "Design docs live in Notion under /engineering/design-docs." The agent doesn't need to memorize the dashboard — it needs to know where to point the user when the topic comes up. 指向信息在外部系统中位置的指针。不是信息本身——只是地址。"流水线 Bug 在 Linear 项目 INGEST 中追踪。""值班延迟仪表盘在 grafana.internal/d/api-latency。""设计文档在 Notion 的 /engineering/design-docs 下。"Agent 不需要记住仪表盘的内容——它需要知道话题出现时该把用户指向哪里。

When to save: When you learn about a resource's location and purpose. Design principle: Reference memories are compact and high-signal. They rarely conflict, rarely go stale (URLs change less often than project state), and provide outsized value relative to their token cost. A 50-character reference memory that saves the user from asking "where do I find the deployment runbook?" pays for itself immediately. 何时保存：了解到资源的位置和用途时。设计原则：Reference 记忆紧凑且高信号。它们很少冲突，很少过期（URL 比项目状态变化更少），且相对 token 成本提供超额价值。一条 50 字符的 Reference 记忆如果能让用户免于问"部署手册在哪里？"，立刻就值回了代价。

Why Four Dimensions, Not One为什么是四个维度，而不是一个

Treating all persistent memory as "preferences" causes three problems. First, retrieval priority becomes flat — a user correction about test strategy competes equally with a Notion URL for context budget, when the correction is clearly more important during a code review. Second, staleness detection is one-size-fits-all — a project deadline from three weeks ago should be treated with suspicion, but a user's role hasn't changed. Third, conflict resolution has no semantics — two feedback memories can conflict ("use mocks" vs. "don't use mocks"), but a feedback memory and a reference memory never do. 将所有持久记忆当作"偏好"会导致三个问题。第一，检索优先级变得扁平——关于测试策略的用户纠正与一个 Notion URL 平等竞争上下文预算，但在代码审查时纠正明显更重要。第二，过期检测一刀切——三周前的项目截止日期应该被怀疑，但用户的角色没变过。第三，冲突解决没有语义——两条 Feedback 记忆可能冲突（"用 mock" vs "不要用 mock"），但 Feedback 记忆和 Reference 记忆永远不会。

The four dimensions give the memory system structured metadata to make better decisions about what to inject, what to verify, and what to discard — without requiring LLM reasoning at the memory layer. 四个维度给记忆系统提供了结构化元数据，使其能更好地决定注入什么、验证什么、丢弃什么——而不需要在记忆层引入 LLM 推理。

A natural question: why not just use Tulving's three-way split (semantic/episodic/procedural) directly? Because agent memory has operational constraints that cognitive science doesn't address. Semantic memory in cognitive science is one pool; in our system, a user's role and a Grafana URL have radically different injection priorities and staleness profiles — lumping them as "semantic" loses operational signal. Similarly, MemGPT's two-tier model (core/archival) is the right architecture, but it doesn't prescribe what goes in each tier. Our four dimensions are the content classification within MemGPT's framework: User and Feedback are core; Project and Reference are contextual. The research gives us the structural scaffolding. The four dimensions give us the operational semantics to fill it. 一个自然的问题：为什么不直接用 Tulving 的三分法（语义/情景/程序）？因为 Agent 记忆有认知科学不涉及的运维约束。认知科学中语义记忆是一个池；在我们的系统中，用户角色和 Grafana URL 的注入优先级和过期特征截然不同——混为"语义"会丢失运维信号。类似地，MemGPT 的两层模型（核心/归档）是正确的架构，但没有规定每层放什么。我们的四个维度是 MemGPT 框架之内的内容分类：User 和 Feedback 是核心层；Project 和 Reference 是情境层。研究提供结构脚手架。四个维度提供填充它的运维语义。

Budget-Aware Context Injection预算感知的上下文注入

All three tiers combine into a single context block that's injected into the system prompt. But context isn't free — every character of memory competes with the user's actual message for context window space. So the injection is budget-constrained. 三层合并为单一上下文块，注入系统提示。但上下文不是免费的——记忆的每个字符都与用户的实际消息竞争上下文窗口空间。因此注入是受预算约束的。

Default budget: 4000 characters. The assembly follows the core/contextual split — core memory (Feedback + User) is always included first, contextual memory (Project + Reference) fills the remaining budget: 默认预算：4000 字符。组装遵循核心/情境分层——核心记忆（Feedback + User）始终最先包含，情境记忆（Project + Reference）填充剩余预算：

  1. Feedback memories — Always included first. These encode hard-won corrections and validated approaches. Missing a feedback memory means repeating a mistake the user already told you about — the fastest way to destroy trust.Feedback 记忆——始终最先包含。它们编码来之不易的纠正和已验证的方法。遗漏 Feedback 记忆意味着重复用户已经告诉你的错误——最快的摧毁信任方式。
  2. User memories — Included next. Role and expertise context is compact and shapes every response. Usually under 200 chars total.User 记忆——其次包含。角色和专业度上下文紧凑且影响每一个回复。通常总共不超过 200 字符。
  3. Project memories — Included if budget allows, most recent first. These provide situational context but go stale, so they're lower priority than user and feedback.Project 记忆——预算允许时包含，最近的优先。它们提供情境上下文但会过期，因此优先级低于 User 和 Feedback。
  4. Reference memories — Included if budget remains. These are compact pointers that add value when relevant but aren't critical for every turn.Reference 记忆——剩余预算时包含。它们是紧凑的指针，相关时有价值但不是每轮都关键。
  5. Conversation — Included from most recent to oldest. If the budget runs out mid-conversation, older turns are dropped.对话——从最近到最旧包含。如果预算在对话中途用完，较旧的轮次被丢弃。
  6. Summaries — Included from most recent, only if budget remains after all above.摘要——从最近的开始包含，仅在上述全部之后预算仍有剩余时。

The LLM never sees the memory machinery. It receives a well-formatted context block that reads like a briefing document: "User preferences: Chinese responses, project root /workspace. Recent context: discussed auth refactoring, completed successfully. Current conversation: [last 6 turns]." The memory system is invisible infrastructure. LLM 永远看不到记忆机制。它接收一个格式良好的上下文块，读起来像简报文档："用户偏好：中文回复，项目根目录 /workspace。近期上下文：讨论了认证重构，成功完成。当前对话：[最近 6 轮]。"记忆系统是不可见的基础设施。

Episode Logging: The Agent's Work JournalEpisode 日志：Agent 的工作日志

Beyond the three tiers, we added a fourth data layer that doesn't feed into prompts: episode logs. Every completed task — whether a single-turn answer or a multi-step refactor — produces a structured record: 在三层之外，我们增加了第四个不注入提示词的数据层：episode 日志。每个完成的任务——无论是单轮回答还是多步骤重构——都产生一条结构化记录： 

{
  "task_type": "team",
  "description": "Refactor task_bot.py into 8 modules",
  "duration_sec": 420,
  "steps_planned": 5,
  "steps_completed": 5,
  "outcome": "success",
  "files_changed": ["config.py", "state.py", "api_client.py", ...],
  "timestamp": "2025-03-15T10:45:00"
}

Episodes don't affect the agent's behavior directly. They exist for offline analysis: Episode 不直接影响 Agent 的行为。它们存在于离线分析：

  • Failure pattern detection — Which task types fail most? Are failures concentrated in specific domains or time windows?失败模式检测——哪类任务失败最多？失败是否集中在特定领域或时间窗口？
  • Duration benchmarking — How long do refactors typically take? Is the agent getting faster over time as system prompts improve?时长基准——重构通常需要多长时间？随着系统提示改进，Agent 是否在变快？
  • Step efficiency — What's the ratio of planned steps to completed steps? A consistently high ratio might indicate over-planning.步骤效率——计划步骤与完成步骤的比率是多少？持续偏高可能表明过度规划。
  • Reflection candidates — Failed episodes with clear root causes become candidates for system prompt improvements or new harness rules.反思候选——有明确根因的失败 episode 成为系统提示改进或新 harness 规则的候选。

Episodes don't feed into prompts — injecting raw episode data would blow the token budget for marginal value. But they're the foundation for something more important: closing the self-improvement loop. These episode metrics also feed directly into the operational evaluation layer — the third layer in a three-tier evaluation system that catches drift, regressions, and silent failures that component tests miss (described in Three Layers of Evaluation). Episode 不注入提示词——注入原始 episode 数据会为边际价值撑爆 token 预算。但它们是更重要的事情的基础：闭合自改进循环。这些 episode 指标也直接输入到运营级评估层——一个三层评估体系中的第三层，用于捕获组件测试遗漏的漂移、回归和静默故障（详见三层评估体系）。

Closing the Loop: From Episodes to Self-Improvement闭合循环：从 Episode 到自改进

Most agent teams improve their system through gut feeling: "The agent seems worse at code reviews lately, let's tweak the prompt." Episode logs replace this with an evidence-driven pipeline. The loop has four stages: 大多数 Agent 团队通过直觉改进系统："Agent 最近代码审查似乎变差了，我们调一下 prompt。"Episode 日志用证据驱动的流水线替代了这种方式。循环有四个阶段：

Stage 1: Pattern Detection阶段一：模式检测

A nightly batch job aggregates episodes from the past 7 days and computes metrics per task type: success rate, average duration, step efficiency (planned vs. completed), and user correction rate (how often the user had to intervene mid-task). The output is a dashboard row, not a neural network — we're looking for trends, not training a model. 每晚批处理任务聚合过去 7 天的 episode 并按任务类型计算指标：成功率、平均时长、步骤效率（计划 vs. 完成）和用户纠正率（用户在任务中途介入的频率）。输出是仪表盘的一行，不是神经网络——我们在找趋势，不是训练模型。

The signals we watch: 我们关注的信号：

  • Success rate drop by task type. If "code refactoring" success rate drops from 85% to 60% over a week, something changed — a new codebase pattern, a model update, or a prompt regression.按任务类型的成功率下降。如果"代码重构"成功率一周内从 85% 降到 60%，说明有什么变了——新的代码库模式、模型更新或 prompt 回退。
  • Duration outliers. A task type that normally takes 3 minutes suddenly averaging 12 minutes suggests the agent is stuck in retry loops or generating excessive intermediate steps.时长异常值。正常 3 分钟的任务类型突然平均 12 分钟，说明 Agent 可能陷入重试循环或生成了过多中间步骤。
  • Correction clustering. If users consistently correct the agent on the same topic (e.g., "stop adding type annotations to Python 3.8 code"), that's a missing feedback memory or a prompt gap — not a one-off mistake.纠正聚集。如果用户在同一话题上反复纠正 Agent（如"别给 Python 3.8 代码加类型注解"），这是缺失的 Feedback 记忆或 prompt 缺口——不是一次性错误。
  • Feedback-to-episode correlation. When a new feedback memory is saved, do subsequent episodes of the same type improve? If not, the feedback isn't being applied effectively — either the injection priority is wrong or the feedback is too vague for the LLM to act on.Feedback 与 episode 的关联。当新的 Feedback 记忆被保存后，同类型的后续 episode 是否改善？如果没有，说明 Feedback 没有被有效应用——要么注入优先级有问题，要么 Feedback 太模糊 LLM 无法执行。

Stage 2: Root Cause Analysis阶段二：根因分析

When a pattern is detected, the next step is not "tweak the prompt." It's "understand why." We pull the failed episodes, read the conversation traces, and classify the root cause: 当检测到模式时，下一步不是"调 prompt"。而是"理解为什么"。我们拉出失败 episode，阅读对话追踪，分类根因：

  • Knowledge gap — The agent doesn't know something it should. Fix: add to system prompt or create a reference memory.知识缺口——Agent 不知道它应该知道的事。修复：添加到系统提示或创建 Reference 记忆。
  • Instruction ambiguity — The prompt says "write clean code" but doesn't define what clean means in this codebase. Fix: make the instruction concrete ("functions under 30 lines, no nested callbacks deeper than 2 levels").指令模糊——Prompt 说"写干净的代码"但没定义在这个代码库中什么是干净。修复：让指令具体化（"函数 30 行以内，回调嵌套不超过 2 层"）。
  • Context starvation — The agent needed information that was evicted from the context window. Fix: adjust budget allocation or promote the information to a higher-priority memory dimension.上下文饥饿——Agent 需要的信息被从上下文窗口驱逐了。修复：调整预算分配或将信息提升到更高优先级的记忆维度。
  • Capability limit — The task is genuinely beyond the model's ability. Fix: add a harness gate that detects this task pattern and escalates to a human instead of letting the agent flail.能力极限——任务确实超出模型能力。修复：添加 Harness 门控检测此任务模式并升级给人类，而不是让 Agent 挣扎。

This step is deliberately human-driven. Automating root cause analysis sounds appealing until you realize it's the step where bad judgment compounds — an incorrect root cause leads to a wrong fix that makes the agent worse. We tried LLM-assisted root cause classification early on. It was right 70% of the time, which sounds acceptable until you realize the 30% wrong fixes silently degraded the system for weeks before someone noticed. 这一步是刻意由人驱动的。自动化根因分析听起来很诱人，直到你意识到这是错误判断会复利的环节——错误的根因导致错误的修复，让 Agent 变得更差。我们早期尝试过 LLM 辅助的根因分类。70% 的时间是对的，听起来可以接受，直到你意识到那 30% 的错误修复悄悄降低了系统质量好几周才被人发现。

Stage 3: Targeted Intervention阶段三：针对性干预

Based on the root cause, we apply the minimum effective change. The intervention types, ordered by preference: 基于根因，我们应用最小有效变更。干预类型按优先级排序：

  1. Add or update a memory. A new feedback memory ("when refactoring Python files, preserve existing import ordering unless explicitly asked to change it") is the lightest intervention. It's per-user, instantly effective, and doesn't require a system-wide change. Most corrections land here.添加或更新记忆。一条新 Feedback 记忆（"重构 Python 文件时保留现有 import 顺序，除非被明确要求更改"）是最轻量的干预。它是按用户的、立即生效的，不需要全系统变更。大多数纠正落在这里。
  2. Update the system prompt. When the same correction appears across multiple users, it graduates from individual feedback memory to a system prompt rule. "Always run the existing test suite before declaring a refactor complete" started as one user's feedback, then appeared independently from three other users within a week. That's a systemic gap, not a personal preference.更新系统提示。当同一纠正出现在多个用户时，它从个人 Feedback 记忆升级为系统提示规则。"在宣布重构完成前始终运行现有测试套件"最初是一个用户的 Feedback，然后一周内从另外三个用户独立出现。这是系统性缺口，不是个人偏好。
  3. Add a harness gate. If the agent consistently fails at a task pattern and the failure has high consequences, add a deterministic check. Example: after three incidents where the agent deleted test files during "cleanup" tasks, we added a gate that blocks file deletion in test directories without explicit user confirmation.添加 Harness 门控。如果 Agent 在某个任务模式上持续失败且后果严重，添加确定性检查。例子：Agent 在"清理"任务中删除测试文件发生三次后，我们添加了一个门控——在没有用户明确确认的情况下阻止删除测试目录中的文件。
  4. Adjust the task routing. Some task types should be reclassified. If "database migration generation" has a 40% success rate despite prompt improvements, maybe it shouldn't be a fully autonomous task — route it to a human-in-the-loop mode where the agent generates a draft and a human reviews before execution.调整任务路由。某些任务类型应该被重新分类。如果"数据库迁移生成"在 prompt 改进后仍然只有 40% 成功率，也许它不应该是完全自主的任务——将其路由到人在环路模式，Agent 生成草稿，人类审查后再执行。

Stage 4: Validation and Rollback阶段四：验证与回滚

Every intervention is treated like a code deploy: change one thing, measure the effect, keep or revert. We maintain a changelog of prompt modifications and memory policy changes, each tagged with the episode pattern that triggered it and the expected metric improvement. 每次干预都像代码部署一样对待：改一件事，衡量效果，保留或回滚。我们维护 prompt 修改和记忆策略变更的变更日志，每条标注触发它的 episode 模式和预期的指标改善。

Validation window: 7 days of episodes post-change. We compare the target metric (e.g., refactoring success rate) against the baseline from the 7 days before the change. If the metric improved, the change stays. If it didn't move or got worse, we revert and re-analyze. No change without measurement, no measurement without a revert plan. 验证窗口：变更后 7 天的 episode。我们将目标指标（如重构成功率）与变更前 7 天的基线对比。如果指标改善，保留变更。如果没有变化或变差，回滚并重新分析。没有度量就没有变更，没有回滚方案就没有度量。

Self-improvement is not autonomous. This is deliberate. The agent doesn't modify its own prompts. Humans analyze the data, decide what to change, make the change, and measure the result. The agent's contribution is generating structured, honest data about its own performance. Autonomous self-modification sounds powerful, but in a production system where the agent handles real user work, an unvalidated prompt change can silently degrade quality for days. We keep the human in the improvement loop for the same reason we keep the human in the execution loop — because the cost of an undetected bad change is much higher than the cost of a human review. 自改进不是自主的。这是刻意为之的。Agent 不修改自己的 prompt。人类分析数据、决定改什么、做出变更、衡量结果。Agent 的贡献是生成关于自身表现的结构化、诚实的数据。自主自修改听起来强大，但在处理真实用户工作的生产系统中，一次未验证的 prompt 变更可以悄悄降低质量好几天。我们在改进循环中保持人类参与，原因和在执行循环中保持人类参与一样——因为未被发现的坏变更的代价远高于人类审查的代价。

What the Loop Looks Like in Practice循环在实践中的样子

A concrete example: episode logs showed that "multi-file refactoring" tasks had a 65% success rate, with most failures caused by the agent missing import updates in dependent files. The root cause was instruction ambiguity — the system prompt said "refactor the target file" but didn't mention transitive dependencies. The fix: a single sentence added to the system prompt — "After modifying a file, check all files that import from it and update references accordingly." Post-change, multi-file refactoring success rate went to 88% within a week. The episode data told us what was failing, the traces told us why, and the 7-day comparison told us the fix worked. 一个具体例子：episode 日志显示"多文件重构"任务成功率 65%，多数失败是因为 Agent 遗漏了依赖文件中的 import 更新。根因是指令模糊——系统提示说"重构目标文件"但没提传递依赖。修复：在系统提示中加一句——"修改文件后，检查所有从它导入的文件并相应更新引用。"变更后一周内，多文件重构成功率升至 88%。Episode 数据告诉我们什么在失败，追踪告诉我们为什么，7 天对比告诉我们修复有效。

Without episode logging, this improvement would have been anecdotal ("I feel like refactors work better now"). With it, the improvement is quantified, attributed, and reversible. That's the difference between engineering and tinkering. 没有 episode 日志，这个改进只是轶事（"我感觉重构现在好多了"）。有了它，改进是量化的、可归因的、可逆的。这就是工程和修补之间的区别。

Failure Modes We've Encountered我们遇到的失败模式

Intra-Dimension Conflicts维度内冲突

Week 1: a feedback memory says "always use mocks for external API calls." Week 3: another feedback memory says "don't mock the database — hit the real instance." Do these conflict? They sound contradictory, but they're about different targets (external APIs vs. database). Without the four-dimension taxonomy, both would land in a flat "preferences" store and the conflict detection would need semantic reasoning. With dimensions, we know both are feedback, and we can apply type-specific conflict rules: feedback memories conflict only when they address the same action in the same scope. The "why" field helps — if the reasons are about different concerns (network reliability vs. migration safety), they don't conflict. 第 1 周：一条 Feedback 记忆说"外部 API 调用始终用 mock"。第 3 周：另一条 Feedback 记忆说"不要 mock 数据库——用真实实例"。它们冲突吗？听起来矛盾，但目标不同（外部 API vs. 数据库）。没有四维度分类，两者会落入扁平的"偏好"存储中，冲突检测需要语义推理。有了维度，我们知道两者都是 Feedback，可以应用类型特定的冲突规则：Feedback 记忆仅在涉及相同范围内的相同操作时才冲突。"Why"字段有帮助——如果原因涉及不同关注点（网络可靠性 vs. 迁移安全性），则不冲突。

For genuinely conflicting memories within the same dimension, the rule is simple: latest timestamp wins. But we only apply this after confirming the scope overlap. This keeps the conflict resolution deterministic and avoidable by the agent without an extra LLM call. 对于同一维度内真正冲突的记忆，规则简单：最新时间戳胜出。但我们只在确认范围重叠后才应用此规则。这使冲突解决保持确定性，Agent 无需额外 LLM 调用即可处理。

Summary Drift摘要漂移

LLM-generated summaries sometimes emphasize the wrong things. A conversation about "refactoring auth with a focus on JWT token rotation" might get summarized as "discussed security improvements" — losing the specific technical detail that would be most useful in a future session. Our mitigation: the summary prompt explicitly asks for "decisions made, tools used, and specific outcomes," not general themes. Structured prompting produces structured summaries. LLM 生成的摘要有时会强调错误的事情。一段关于"重构认证，聚焦 JWT token 轮换"的对话可能被总结为"讨论了安全改进"——丢失了在未来会话中最有用的具体技术细节。我们的缓解：摘要提示明确要求"做出的决策、使用的工具和具体结果"，而非一般性主题。结构化提示产生结构化摘要。

Token Budget StarvationToken 预算饥饿

Early on, we set the budget too low (2000 chars). Result: after preferences and 2-3 conversation turns, there was no room left for summaries. The agent had good short-term memory and good preference recall, but zero awareness of past sessions. We doubled the budget to 4000 chars. The extra 2000 chars go almost entirely to summaries, which dramatically improved the "we discussed this last week" experience. The cost: slightly longer system prompts. The benefit: users stopped repeating themselves across sessions. 早期我们把预算设得太低（2000 字符）。结果：在偏好和 2-3 轮对话之后，摘要没有空间了。Agent 有良好的短期记忆和偏好召回，但对过去会话零感知。我们将预算翻倍到 4000 字符。额外的 2000 字符几乎全部给了摘要，这极大改善了"我们上周讨论过这个"的体验。代价：略长的系统提示。收益：用户不再跨会话重复自己。

Storage Design存储设计

Each user gets a directory on disk: 每个用户在磁盘上有一个目录： 

.state/memory/{user_id}/
├── conversation.json   # sliding window (hot, in-memory + periodic flush)
├── user.json           # role, expertise, context (write-through)
├── feedback.json       # corrections + validated approaches (write-through)
├── project.json        # ongoing work, deadlines, decisions (write-through)
├── reference.json      # pointers to external resources (write-through)
├── summaries.json      # compressed past sessions (append-only)
└── episodes.json       # task completion records (append-only)

All writes use atomic rename (write to .tmp, then rename) to prevent corruption on crash. The conversation tier is primarily in-memory with periodic flushes; the four persistent dimensions and summaries are write-through. On process restart, the system loads all files and resumes with full memory intact. Splitting persistent memory into four files instead of one monolithic preferences.json also makes manual inspection trivial — cat feedback.json shows exactly what the agent has learned from corrections, without scrolling past user profile data and project deadlines. 所有写入使用原子重命名（写入 .tmp，然后 rename）以防止崩溃时损坏。对话层主要在内存中，定期刷新；四个持久维度和摘要是写穿的。进程重启时，系统加载所有文件，完整记忆恢复运行。将持久记忆拆分为四个文件而非一个单体 preferences.json 也使手动检查变得简单——cat feedback.json 可以只看 Agent 从纠正中学到了什么，而不用翻过用户画像数据和项目截止日期。

Why flat files and not a database? Because the access pattern is simple (read-all-on-startup, append-on-event, read-by-user-id), the data volume is small (kilobytes per user), and file-based storage is trivially portable and debuggable. You can cat preferences.json to see exactly what the agent remembers. No ORM, no migration scripts, no connection pooling. The right tool for the scale. 为什么用文件而不是数据库？因为访问模式简单（启动时全量读取、事件时追加、按用户 ID 读取），数据量小（每用户千字节级），文件存储天然可移植和可调试。你可以 cat preferences.json 来看 Agent 到底记住了什么。不需要 ORM、迁移脚本、连接池。适合当前规模的正确工具。

Design Principles设计原则

Five principles emerged from building and operating this system: 构建和运营这个系统后浮现出五条原则：

  1. Different information, different lifetimes and types. Don't treat all memory equally. Following MemGPT's core/archival principle: classify persistent memories into core (Feedback + User — always injected, hard size cap) and contextual (Project + Reference — budget-dependent, staleness-managed). Conversations are ephemeral. Summaries are archival. Design storage, eviction, and injection policies based on the cognitive science insight that semantic, procedural, and episodic memory serve fundamentally different retrieval needs.不同的信息，不同的生命周期和类型。不要同等对待所有记忆。遵循 MemGPT 的核心/归档原则：将持久记忆分为核心（Feedback + User——始终注入、硬性大小上限）和情境（Project + Reference——依赖预算、需过期管理）。对话是短暂的。摘要是归档的。基于认知科学的洞察来设计存储、驱逐和注入策略：语义、程序和情景记忆服务于根本不同的检索需求。
  2. Budget beats completeness. It's better to inject 4000 characters of high-priority memory than 20000 characters of everything. The LLM performs worse with bloated context, and you pay for every token. Curate aggressively.预算优于完整性。注入 4000 字符的高优先级记忆比注入 20000 字符的所有内容更好。LLM 在膨胀的上下文中表现更差，而且你为每个 token 付费。积极策展。
  3. Memory is invisible infrastructure. The LLM should never see memory management commands or tier labels. It receives a context block that looks like a briefing. The machinery is human-side, not model-side.记忆是不可见的基础设施。LLM 不应该看到记忆管理命令或层级标签。它接收一个看起来像简报的上下文块。机制是人侧的，不是模型侧的。
  4. Crash-safe by default. Every state transition writes to disk atomically. The system can be killed at any point and restart with full memory. This isn't optional for production agents — it's table stakes.默认崩溃安全。每次状态转换原子写入磁盘。系统可以在任何点被杀死并以完整记忆重启。对于生产 Agent 这不是可选的——这是基本要求。
  5. Observe before you optimize. Episode logs exist to make the memory system's impact measurable. Without them, "the agent seems to remember better" is a feeling, not a fact. With them, you can answer: how often do users repeat preferences? How many tasks reference past session context? Is the summary quality improving?先观察再优化。Episode 日志的存在是为了使记忆系统的影响可衡量。没有它们，"Agent 似乎记忆更好了"是一种感觉，不是事实。有了它们，你可以回答：用户多频繁重复偏好？多少任务引用了过去会话的上下文？摘要质量在改善吗？
We've been running this memory system for eight months. The question we still can't answer: how do you evaluate memory quality? We can measure whether the agent uses injected memories — that's just prompt tracing. We can measure whether users repeat themselves less — that's a proxy for recall. But we can't easily measure whether the agent would have been better off without a stale memory. A project deadline from two months ago that's still in context doesn't cause a visible error — it causes a subtly wrong assumption that might never surface. We track memory injection rate, user-repetition rate, and correction frequency. We still don't have a metric for "memory that's actively making things worse by being present." If you figure that one out, let me know. 这个记忆系统已经运行了八个月。我们仍然无法回答的问题是：如何评估记忆质量？我们可以衡量 Agent 是否使用了注入的记忆——那只是 prompt 追踪。我们可以衡量用户是否减少了重复——那是召回的代理指标。但我们无法轻易衡量 Agent 是否在没有一条过期记忆的情况下会表现更好。一个两个月前的项目截止日期仍然在上下文中不会导致可见的错误——它导致一个微妙的错误假设，可能永远不会浮出水面。我们追踪记忆注入率、用户重复率和纠正频率。我们仍然没有一个"因为存在而正在积极恶化结果的记忆"的指标。如果你找到了这个答案，告诉我。

Checklist: Building Memory for Your Agent清单：为你的 Agent 构建记忆

If you're adding memory to an agent system, here's where to start. Each item is ordered by impact — the first few deliver most of the value. 如果你正在为 Agent 系统添加记忆，以下是起步指南。按影响力排序——前几项带来最大价值。

  1. Separate persistent memory from conversation. At minimum, extract user preferences ("reply in Chinese," "always use /workspace") into a file that survives session resets. This single change eliminates the most common trust-eroding behavior: the agent forgetting what you already told it.将持久记忆与对话分离。至少把用户偏好（"用中文回复"、"始终使用 /workspace"）提取到一个会话重置后仍保留的文件中。这个单一变更消除了最常见的信任侵蚀行为：Agent 忘记你已经告诉它的事情。
  2. Add session TTL with compression. When the user is inactive for a threshold period (start with one hour), compress the conversation into a one-paragraph summary before evicting it. This preserves cross-session continuity without carrying stale context.添加带压缩的会话 TTL。当用户不活跃超过阈值（从一小时开始），在驱逐对话前将其压缩为一段摘要。这在不携带陈旧上下文的情况下保持跨会话连续性。
  3. Classify persistent memories into core and contextual. At minimum, separate user corrections (feedback — procedural memory) from general preferences (semantic memory), and guarantee core memories are always injected. Feedback memories encode hard-won lessons and should always come first. The full taxonomy (user, feedback, project, reference) maps to established models: Tulving's semantic/procedural/episodic distinction for content types, MemGPT's core/archival distinction for injection policy.将持久记忆分为核心和情境两级。至少把用户纠正（feedback——程序记忆）和一般偏好（语义记忆）分开，并保证核心记忆始终被注入。Feedback 记忆编码来之不易的教训，应始终最先注入。完整分类（user、feedback、project、reference）映射到成熟模型：Tulving 的语义/程序/情景区分用于内容类型，MemGPT 的核心/归档区分用于注入策略。
  4. Set a context budget and enforce priorities. Pick a character budget (start with 4000 characters) and fill it in priority order: feedback first, then user profile, then project context, then conversation turns. Never let conversation history crowd out persistent memories.设定上下文预算并执行优先级。选择一个字符预算（从 4000 字符开始），按优先级填充：先 feedback，然后用户画像，然后项目上下文，最后对话轮次。永远不要让对话历史挤掉持久记忆。
  5. Use atomic writes for all state. Write to a temp file, then rename. This prevents corruption on crash and makes your memory system restart-safe.所有状态使用原子写入。先写临时文件，然后重命名。这防止崩溃时损坏，使记忆系统重启安全。
  6. Log episodes for every completed task. Record task type, duration, step count, outcome, and files changed. You don't need to inject this into prompts — it's for offline analysis. Without it, you can't measure whether your memory system is actually improving agent performance.为每个完成的任务记录 episode。记录任务类型、时长、步骤数、结果和更改的文件。不需要注入到提示词中——它是用于离线分析的。没有它，你无法衡量记忆系统是否真正改善了 Agent 表现。
  7. Keep self-improvement human-driven. Use episode data to identify failure patterns, but have humans decide what to change. Automated root cause analysis sounds appealing but compounds errors — a 70% accuracy rate on fixes means 30% of changes silently degrade your system.保持自改进由人驱动。使用 episode 数据识别失败模式，但由人决定改什么。自动化根因分析听起来诱人但会复利错误——修复准确率 70% 意味着 30% 的变更在悄悄降低系统质量。

Frequently Asked Questions常见问题

How much of the context budget should memory injection consume?记忆注入应该占用多少上下文预算？

We found 4000 characters to be the practical sweet spot for total memory injection. Core memories (Feedback + User) typically consume under 700 characters combined, leaving 3300+ characters for contextual memories, conversation turns, and summaries. Going below 3000 characters starves summaries entirely; above 6000 characters inflates system prompts without measurable benefit because the LLM's attention to injected context degrades with length.我们发现 4000 字符是记忆注入总量的实际最优点。核心记忆（Feedback + User）通常合计不超过 700 字符，留下 3300+ 字符给情境记忆、对话轮次和摘要。低于 3000 字符会完全饿死摘要；高于 6000 字符会膨胀系统提示但没有可衡量的收益，因为 LLM 对注入上下文的注意力会随长度衰减。

How do you prevent the agent from hallucinating memories it was never given?如何防止 Agent 幻觉出从未被给予的记忆？

Memory injection is append-only to the system prompt — the agent cannot query or modify the memory store. All memories are presented as a structured briefing block with explicit labels, so the model treats them as given context rather than self-generated knowledge. We also never use phrasing like "you remember that..." in the injection; instead we use "User has stated:" or "Prior correction:" to make the provenance unambiguous. In eight months of operation, fabricated-memory incidents have been near zero with this framing.记忆注入是系统提示的仅追加块——Agent 无法查询或修改记忆存储。所有记忆以带有明确标签的结构化简报块呈现，使模型将其视为给定上下文而非自生知识。我们也从不在注入中使用"你记得..."这样的措辞；而是用"用户已声明："或"先前纠正："来使来源无歧义。在八个月运营中，这种框架下虚构记忆事件几乎为零。

When should you migrate from flat-file storage to a database?什么时候应该从文件存储迁移到数据库？

The trigger is concurrent writes, not data volume. Flat files work well up to hundreds of users because each user's data is isolated to their own directory and the read-all-on-startup pattern stays sub-millisecond. The breakpoint comes when multiple agent processes serve the same user simultaneously — file-level locking becomes fragile. At that point, move to SQLite (single-writer, still file-based, zero-config) before considering PostgreSQL. We have not needed to make this migration yet at our current scale.迁移触发条件是并发写入，而非数据量。文件存储在数百用户以下工作良好，因为每个用户的数据隔离在自己的目录中，启动时全量读取模式保持亚毫秒。临界点出现在多个 Agent 进程同时服务同一用户时——文件级锁定变得脆弱。此时先迁移到 SQLite（单写入者、仍基于文件、零配置），再考虑 PostgreSQL。我们在当前规模下尚未需要进行此迁移。

Can this memory architecture work with multi-agent systems where several agents share context?这个记忆架构能用于多个 Agent 共享上下文的多 Agent 系统吗？

Yes, but you need a scoping layer. In a multi-agent setup, User and Feedback memories should be scoped to the individual agent-user pair because different agents may receive contradictory corrections for legitimate reasons (e.g., a coding agent and a review agent have different "don't do this" rules). Project and Reference memories can be shared across agents since they describe the environment, not agent behavior. The injection budget must also be per-agent — sharing a single 4000-character budget across three agents leaves each one context-starved.可以，但需要一个作用域层。在多 Agent 设置中，User 和 Feedback 记忆应该限定在单个 Agent-用户对，因为不同 Agent 可能出于正当原因收到矛盾的纠正（例如编码 Agent 和审查 Agent 有不同的"不要这样做"规则）。Project 和 Reference 记忆可以跨 Agent 共享，因为它们描述环境而非 Agent 行为。注入预算也必须是每个 Agent 独立的——三个 Agent 共享单一的 4000 字符预算会导致每个都上下文饥饿。
